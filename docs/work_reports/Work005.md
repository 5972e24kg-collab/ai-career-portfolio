# Work005: 次世代自宅LLM基盤のハードウェア選定

Tag: [[Hardware]], [[GMKtec]], [[LLM]], [[Architecture]], [[SystemDesign]]

## 1. 今回の作業の目的 [#purpose]

#contents

* 現在のハードウェア環境（RTX 4060 8GB）における「VRAM不足によるモデルサイズの制約」と「RAG更新時の不安定さ」を根本解決する。
* 70B〜100B超クラスのLLMや、高度なマルチエージェント環境をローカルで稼働させる可能性を広げるための「大容量メモリ」を確保する。
* 「速度」よりも「容量」と「省スペース性」を優先し、既存環境と共存するハイブリッドな自宅サーバー基盤を設計する。

## 2. システム環境（移行計画） [#environment]

* **Current Node (Node A):**

  * **Hardware:** NVIDIA GeForce RTX 4060 (VRAM 8GB)
  * **Role:** 軽量・高速推論 (Whisper, Stable Diffusion), CUDA依存タスク, Python/Java ETL処理
* **New Node (Node B):**

  * **Hardware:** GMKtec EVO-X2 (Ryzen AI Max+ 395 / RAM 128GB)
  * **Role:** 大規模推論 (70B+ LLM), RAGコンテキスト保持, コーディングエージェント, マルチモーダル統合
  * **Memory Allocation:** GPU割当96GBを想定（購入検討時点） + System 32GB

## 4. 作業の軌跡と解決プロセス [#process]

### 4.1 発生した課題・検討事項 [#issues]

* **VRAM 8GBの限界:** 当時の構成では14B級モデル付近がRAG運用における実用上の上限となっており、期待していたコーディング支援や複雑な推論では十分な品質を得にくかった。
* **市場のジレンマ:** RTX 3090/4090等のハイエンドGPUは高価かつ巨大で、自宅サーバーとしての常時稼働（消費電力・騒音）に課題があった。
* **DRAM価格の高騰:** AI需要を背景としたメモリ価格上昇の可能性を考慮し、大容量マシンの価格や入手性が悪化するリスクを懸念した。

### 4.2 試行錯誤のプロセス（選定フロー） [#selection]

1. **RTX 50/40シリーズの検討** -> VRAM容量が24GB〜32GBに留まり、今回の目的である「容量の壁」を十分に突破できないため却下。
2. **ミニPC (Ryzen 5000系/Intel Core Ultra) の検討** -> 検討した機種では、LLM用途に必要と考えたメモリ容量・メモリ帯域・iGPU性能の要件に届かないと判断。
3. **GMKtec EVO-X2 (Ryzen AI Max+) の発見** -> Mac Studioと同様にCPUとGPUで大容量メモリを共有するユニファイドメモリアーキテクチャを持ち、Windowsで動作する点に着目。
4. **64GB vs 128GB の比較** -> メモリ増設不可の仕様と、将来的な「100B超モデル」の搭載可能性を見据え、4.7万円差なら128GBを選ぶべきと判断。
5. **ネガティブ要素の検証 (VS RTX 3090)**

* 「CUDAが使えない」「RTX 3090と比較するとメモリ帯域や推論速度で不利になる可能性がある」という欠点を、既存のRTX 4060との「ハイブリッド構成」で補完する戦略を立案。
* 「速度を妥協してでも、同価格帯の単体GPU構成では確保しにくい『容量』を手に入れる」という明確なトレードオフを選択。

### 4.3 解決策・決定打 [#decision]

* **決定打:** **「DRAM価格の上昇リスクを踏まえた128GBメモリの確保」**と**「ハイブリッド・アーキテクチャの確立」**。
* **戦略:**

  * **Node A (RTX 4060):** 「手足」として、軽い処理やCUDA必須の処理を高速にこなす。
  * **Node B (EVO-X2):** 「脳」として、70B級モデルによる大規模推論や長いコンテキストの処理を担当する。
* これにより、単一のマシンですべてを解決しようとする無理を排除し、処理特性に応じて役割を分担する環境を構築する方針を固めた。
