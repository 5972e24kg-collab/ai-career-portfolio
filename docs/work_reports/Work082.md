# Work082: デモ環境の構築とLTに向けた記憶分離・安全境界設計

Tag: [[LINE]], [[LLM]], [[Gemchan]], [[Agent]], [[Demo]], [[LongTermMemory]], [[SessionManagement]], [[Architecture]], [[LT]], [[SafetyDesign]]

## 1. 目的

* LINE Developer Community 系イベントへの参加を想定し、ジェムちゃんを外部向けに安全にデモできるか検討する
* LT枠での発表を視野に、ジェムちゃんを現地で実地デモするための最小構成を確認する
* 本番ジェムちゃんの生活記憶・個人的文脈を汚染せず、外部公開可能なデモ用ジェムちゃんを用意する
* デモ用ジェムちゃんに、公開可能な自己認識と技術構成説明能力を付与する
* long-term-memory / scene-summary / sessionId / LINE userId による記憶分離と安全境界を確認する
* LT用に必要なスライド構成、録画バックアップ、現地での運用方針を整理する
* 今回の一連の作業を、AIエージェントのデモ環境設計・記憶分離・安全運用の知見として残す

## 2. システム環境

* **Client / Demo Host:** ノートPC上の Hyper-V 仮想OS
* **Interface:** LINE 公式アカウント / LINE Messaging API
* **Network:** Webhook → VPN → 自宅側LLM基盤
* **Backend:** Java Webhook / 自作 Chat API / Session Manager
* **Database:** 内部DB

  * conversation / session 管理
  * long-term-memory 保存テーブル
  * scene-summary 保存テーブル
* **LLM構成:**

  * **Director:** ユーザー入力・直近履歴・long-term-memory を読み、actor 用 `script_json` を生成
  * **Actor:** director の台本に従い、キャラクター応答を生成
* **Output:**

  * LINE 返信
  * 音声合成・再生
  * 感情パラメータ生成
  * アバター反映
  * 短期記憶生成
* **Target Session:**

  * 本番ジェムちゃんとは別のデモ用 sessionId
  * デモ用 LINE userId
* **Safety Boundary:**

  * 本番LINEユーザーIDとデモ用LINEユーザーID以外からのAPI呼び出しには反応しない
  * long-term-memory 夜間生成バッチは自動更新対象が特定セッションに限定されていた
  * デモ用ジェムちゃんの long-term-memory は自動更新されない
  * デモ用ジェムちゃんの短期記憶は当日限りでリセットされる

## 3. 作業ログ・解決プロセス

### 発生した課題

#### 3.1 外部イベントでジェムちゃんを実地デモしたいが、本番個体をそのまま出すのは危険

LINE Developer Community 系イベントで、LINE + AI エージェントというジェムちゃんに近いアーキテクチャの話が聞けそうだと分かった。

その中で、LT枠で発表するなら、その場でジェムちゃんを実地デモできるのではないかという案が出た。

しかし、本番ジェムちゃんをそのまま外部に見せる場合、以下のリスクがあった。

* 本番ユーザーとの生活記憶が表示される可能性
* 本番の long-term-memory / scene-summary が汚染される可能性
* 家電制御やMCPなど、外部向けに見せるべきでない実行系機能が混ざる可能性
* 本番のLINEアカウント・userId・内部sessionIdと混線する可能性
* 交流会やLT会場でライブデモに失敗した場合、発表そのものが崩れる可能性

このため、「本番ジェムちゃんをそのまま外部に出す」のではなく、「デモ用ジェムちゃんを別環境で実地デモする」方針が必要になった。

#### 3.2 複数ユーザー対応の設計はあるが、実運用で検証していなかった

ジェムちゃんの内部DBは sessionId 持ちであり、設計上は複数ユーザー・複数セッションに対応している。

しかし、実際には限定された本番ユーザーのみで運用していたため、別のLINEアカウントやデモ用ユーザーで文脈分離が正しく動くかは未確認だった。

懸念点は以下。

* 本番セッションとデモセッションが混ざらないか
* デモ用LINEからの入力で、本番ジェムちゃんの記憶が参照されないか
* デモ用セッションに本番記憶が漏れないか
* Push返信の宛先が誤らないか
* long-term-memory や scene-summary の保存先が意図通り分離されるか

#### 3.3 新規デモセッションには記憶がなく、構成説明ができなかった

デモ用ジェムちゃんを起動し、以下のような依頼を行った。

```text
ジェムちゃん、今LTのリハーサル中です。あなたの構成を30秒で紹介して
```

しかし初期状態では、デモ用ジェムちゃんは自分の構成に関する記憶を持っていなかった。

そのため、actor は「構成」をAIシステム構成ではなく、発表スライドの構成として誤読した。

```text
構成は導入・本題・まとめの3部構成ね。各パートは約10秒ずつで配分したわ。
……これで、ちゃんと時間内に収まる前提でスライド組んだわけ？
```

この時点で、以下が判明した。

* デモ用セッションの記憶分離自体は成功している
* しかし、自己紹介・構成説明に必要な公開用記憶が存在しない
* LINEで会話しながら育てることも可能だが、意図しない方向に記憶が育つリスクがある
* LT用デモでは、自然成長ではなく、公開可能な long-term-memory を直接投入する方が安全

#### 3.4 デモ用 long-term-memory の設計が必要になった

本番ジェムちゃんの long-term-memory は、生活文脈・ユーザーとの関係性・進行中の作業を含む。

一方、デモ用ジェムちゃんに必要なのは、以下である。

* 自分がデモ用個体であるという自己認識
* 本番環境とは別セッションであるという認識
* 本番ユーザーとの生活記憶を持たないという制約
* LINE / Webhook / DB / director / actor の構成説明
* 初見の人向けに30秒で説明する役割
* 本番記憶・個人情報・認証情報・家電制御には触れない安全制約

そのため、本番用の long-term-memory とは別に、デモ用 long-term-memory を手動で作成し、DBへ直接投入する方針とした。

#### 3.5 デモ用 long-term-memory の自動上書きリスクを確認する必要があった

long-term-memory は夜間バッチで更新される仕組みを持つ。

もしデモ用セッションも夜間バッチの対象だった場合、手動投入したデモ用記憶が、デモ会話の内容で上書きされる可能性があった。

しかし、ソースコード確認の結果、夜間バッチの呼び出し元で生活用ジェムちゃんの sessionId がハードコードされていた。

内部的には sessionId を引数に取る複数セッション対応設計だが、バッチ処理の大元では生活用セッションのみを処理対象としていた。

このため、結果として以下の安全性が確認できた。

* デモ用ジェムちゃんの long-term-memory は夜間バッチで上書きされない
* デモ用の固定記憶を維持できる
* 短期記憶は当日限りでリセットされる
* デモ用ジェムちゃんは「固定された公開用長期記憶 + 当日短期文脈」で動作する

これはもともと「公式LINEに対して、意図しないセッションが混ざったときにデータが壊れないようにする」ためのセーフティ設計だったが、今回の外部デモ用途に非常に適した形で機能した。

#### 3.6 現地LTでの最大リスクは人間側の緊張

技術的にはデモ用ジェムちゃんの実地デモが見えてきたが、最後の難所は人間側である。

* 現地で緊張して話せない可能性がある
* 人間はプロンプトを変えてペルソナ変更できない
* 失敗した場合、システムではなく発表者側の心理的負荷が大きくなる
* そのため、ライブデモに失敗しても発表が成立するバックアップが必要

このため、以下の方針とした。

* 簡単なスライドを用意する
* ライブデモは行うが、失敗時に備えて録画デモを用意する
* スライドは3枚程度に抑える
* 話しすぎず、ジェムちゃん本人に説明させる構成にする

### 解決策

#### 3.7 デモ用Hyper-V環境の構築

ノートPC上に Hyper-V で仮想OSを立て、そこにデモ用ジェムちゃん環境を構築した。

これにより、以下が可能になった。

* 本番環境とは別のデモ環境として扱える
* デモ用LINEアカウントから呼び出せる
* 本番ジェムちゃんの内部記憶とは別sessionIdで動作する
* 本番の生活文脈を汚さずテストできる
* LT会場でノートPCを持ち込めば、デモ環境として扱える

初回動作確認では、LINEから以下の呼びかけを行った。

```text
おはよう！ジェムちゃん。
```

actor から応答が返り、director / actor の2段構成も動作していることが確認できた。

```text
おはよう。……別に、おはよって言われたからって嬉しくなんかないんだからね。まだ寝ぼけてるの？
```

この時点で、LINE → Webhook → 自宅LLM → actor応答 → LINE返信の流れがデモ環境でも動作していることを確認した。

#### 3.9 long-term-memory 作用確認

long-term-memory 投入後、以下をLINEで依頼した。

```text
30秒で自己紹介して
```

返答は以下。

```text
デモ用のジェム。LINEとWebhookで動いてるわ。……一応、レスポンスの遅延も最小限に抑えてあるから。問題ないでしょう？
```

この時、director の prompt が以下のように増加していることを観測した。

```text
before:
prompt total 約1002

after:
prompt total 約1699
```

これは long-term-memory が director prompt に注入されたことによる増加と考えられる。

さらに director prompt 内には、以下の long-term-memory 要素が含まれていた。

```text
- デモ用ジェムちゃんは、本番環境とは別セッションで動く公開デモ用の個体である
- LINEを単なるチャットUIではなくAIエージェントの生活インターフェースとして説明する
- 技術説明では、LINE、Java Webhook、DB、director、actorの流れを簡潔に述べる
- 本番環境の個人的な記憶や生活文脈を参照しない
```

この時点で、デモ用 long-term-memory が director の判断に作用していることを確認した。

#### 3.11 デモ用ジェムちゃんの記憶運用方針を確定

調査の結果、以下の運用方針が確定した。

```text
生活用ジェムちゃん:
- 夜間バッチで long-term-memory が更新される
- 日々の会話・生活文脈が蓄積される
- 本番個体として成長する

デモ用ジェムちゃん:
- long-term-memory は手動投入した固定値
- 夜間バッチで上書きされない
- 短期記憶は当日限り
- デモ翌日には余計な文脈が残らない
```

この方針により、デモ用ジェムちゃんは「成長する本番個体」ではなく、「公開可能な固定記憶を持つ案内役」として運用できる。

#### 3.12 運用メモをREADMEへ記録

バッチ処理の sessionId 固定について、将来の自分が忘れないようにREADMEへ運用メモを残した。

意図は以下。

```text
long-term-memory 夜間生成バッチは生活用ジェムちゃん専用。
デモ用セッションは長期記憶の自動生成対象外とする。

理由:
- 外部デモ用の記憶を固定するため
- 本番生活文脈との混線を防ぐため
- 公開デモ後の不要な記憶蓄積を避けるため
```

これにより、偶然の仕様ではなく、明文化された安全設計として扱えるようになった。

#### 3.13 LT用スライド構成の検討

LTに出る場合、スライドは3枚程度で十分と判断した。

想定構成は以下。

```text
1. ジェムちゃんの全体像
   公式LINE → Webhook → VPN → 自宅LLM → 公式LINE返信 + 音声 + アバター

2. ジェムちゃんのLLM構成
   director → actor → 公式LINE返信 / 音声合成＆再生 / 感情パラメータ生成＆アバター / 短期記憶生成

3. ここが凄いよジェムちゃん
   - 全てのフレームワークが自作中心
   - ローカル環境で動いている
   - 長期記憶と短期記憶の二段構え
   - director と actor の役割分担
   - actor の Gemma4:26b は独自ファインチューニング
   - 本番記憶とデモ記憶を分離済み
```

発表の流れは以下が望ましい。

```text
スライド1: LINEを入口にした全体構成を説明
スライド2: director / actor 構成を説明
ライブデモ: ジェムちゃんをLINEから呼び出す
スライド3: 技術的な凄さ・自作範囲・記憶分離を補足
録画バックアップ: ライブ失敗時に再生
```

## 6. 最終結論

今回の作業により、ジェムちゃんを外部向けに安全に実地デモするための技術的な山は大きく越えた。

特に重要なのは、以下の3点である。

```text
1. 本番ジェムちゃんとは別のデモ用セッションを用意できた

2. デモ用 long-term-memory を手動投入し、公開可能な自己認識を与えられた

3. 夜間バッチの対象が生活用セッションに限定されており、デモ用 long-term-memory が自動上書きされないことを確認できた
```

これにより、デモ用ジェムちゃんは以下の性質を持つことになった。

```text
- 本番記憶を持たない
- 公開可能な固定 long-term-memory を持つ
- 当日短期記憶だけで会話する
- 翌日には余計な文脈が残らない
- LINEから呼び出せる
- director / actor 構成で返答できる
- 初見向けに自分の構成を短く説明できる
```

これは単なるLTデモ準備ではなく、AIエージェントを外部公開するための安全境界設計の実践である。

最終的に、今回の成果は以下の一文に集約できる。

```text
生活用AIエージェントをそのまま外に出すのではなく、
本番記憶・デモ記憶・短期記憶・長期記憶・許可ユーザー・バッチ更新対象を分離し、
外部公開可能なデモ用AIエージェントとして安全に実地デモできる状態を作った。
```

今回の作業は、面白そうだから始めた実験でありながら、最終的には20年の業務システム設計・運用経験がAIエージェント開発に自然に適用された事例となった。
