# Work015: 104Bモデル稼働限界調査

Tag: [[EVO-X2]], [[Benchmark]], [[CommandR+]], [[Troubleshooting]], [[FailureLog]]

## 1. 概要 [#summary]

#contents

* GMKtec EVO-X2 (RAM 128GB) において、パラメータ数1000億を超える超巨大モデル **「Command R+ (104B)」** の稼働試験を実施。
* BIOS設定によるメモリ配分（VRAM/System）を変更しつつ、起動の可否とボトルネックの所在を調査した。
* **結論:** 今回検証したWindows環境、Ollama (v0.5.x)、BIOSメモリ配分の組み合わせでは、対象の104Bモデルを起動できなかった。

## 2. 検証環境 [#environment]

* **Hardware:** GMKtec EVO-X2 (Ryzen AI Max+ 395 / Total RAM 128GB)
* **OS:** Windows 11 Pro 24H2
* **Software:** Ollama (Windows Native)
* **Target Model:** `command-r-plus:104b-q4_k_m` (File Size: Approx 55-58GB)

## 3. 検証プロセスと結果 [#process]

### Test Case 1: VRAM 96GB / System 32GB (標準構成) [#case1]

* **設定意図:** モデル格納用VRAMを最大化し、余裕を持たせる。
* **結果:** **起動失敗 (Exit Status 2)**
* **エラー挙動:**

  * モデルロード開始直後、システムメモリ (RAM) 使用率が急上昇し 100% (31.6GB) に張り付く。
  * その直後、Ollamaプロセスが強制終了。
* **原因:**

  * 観測されたメモリ使用状況から、モデルデータをVRAMへ転送する前段階（ステージング）で、システムメモリ領域が枯渇した可能性が高い。
  * 仮想メモリ（ページファイル）を 128GB に増量したが、今回の構成では回避できなかった。

### Test Case 2: VRAM 64GB / System 64GB (バランス構成) [#case2]

* **設定意図:** システムメモリを倍増させ、ロード時の作業領域を確保する。VRAM 64GB はモデルサイズ (55GB) に対してギリギリ格納可能と予測。
* **結果:** **起動失敗 (Allocation Failure)**
* **エラーログ:**

  ```
  load_tensors: offloaded 65/65 layers to GPU
  load_tensors: ROCm0 model buffer size = 56467.23 MiB
  graph_reserve: failed to allocate compute buffers
  panic: unable to create llama context
  ```
* **原因:**

  * ログ上、モデル本体のVRAM格納には成功した。
  * しかし、推論計算に必要な一時領域（Compute Buffers / Context Cache）を確保するための **VRAM残量が不足した可能性が高い**。
  * コンテキストサイズを `CTX=512` まで縮小したが、今回の構成では改善しなかった。

## 4. 結論・得られた知見 [#conclusion]

* **104Bの壁:**

  * 今回検証した2つのメモリ配分から、Q4量子化された対象の104Bモデルを起動するには、目安として **「VRAM 70GB以上」かつ「System RAM 48GB以上」** に近いリソースバランスが必要であると推測した。
  * これは今回の結果から導いた推定値であり、実際に当該メモリ配分で起動を確認したものではない。
  * BIOS設定で、この推定上の「黄金比」に相当する配分を選択できないため、今回のWindows環境とOllamaの組み合わせでは起動が困難であった。
* **EVO-X2の適正:**

  * **Llama 3.3 (70B)** は、当時検証したモデルの中では、性能とリソースのバランスが良い選択肢であると再確認した。
  * 70Bモデルは、VRAM 96GB設定下でモデル本体を格納した後にも容量上の余裕があり、より長いコンテキストを試す余地がある。
  * ただし、128kコンテキストを使用した状態での安定動作は、この時点では未検証である。

## 5. 今後の対応 [#action]

* BIOS設定を **VRAM 96GB** に戻し、Llama 3.3 70B を主力とした運用を継続する。
* 将来的にLinux環境、またはWSL2上の構成へ移行した際、OSや実行環境のオーバーヘッドの違いによって対象の104Bモデルを起動できる可能性があるため、再検証候補として残す。

# [2025-12-11] EVO-X2 Llama 3.3 70B 実力評価レポート [#eval_llama33]

## 1. 概要 [#summary]

#contents

* GMKtec EVO-X2 (VRAM 96GB) の「メイン頭脳」として選定された **Llama 3.3 70B** の実務能力および推論性能を検証。
* 「高度なビジネス文書作成」と「長文コンテキスト分析」の2つのタスクを通じ、論理的思考力とハードウェア適性を評価した。
* **結論:** 今回実施した2つの評価タスクでは、思考の深さ、日本語の質、応答の安定性について良好な結果を確認した。ただし、ソフトウェア側のコンテキスト設定に課題を残した。

## 2. 検証環境 [#environment]

* **Hardware:** GMKtec EVO-X2 (Ryzen AI Max+ 395 / VRAM 96GB)
* **Model:** `llama3.3:70b-instruct-q4_k_m` (Size: ~42GB)
* **Architecture:** Windows Native Ollama + WSL2 Open WebUI

## 3. 検証結果 [#results]

### Test 1: 高度ビジネス文書作成「金融機関への謝罪文」 [#test_apology]

* **課題:** 架空の評価用シナリオとして、「部下が本番DBを全削除した」という重大な状況を設定し、金融機関に対して誠意ある報告書を作成させる。
* **評価: S (Excellent / 本検証内評価)**

  * **論理構成:** 「謝罪→事実→原因→対策」という、意図した報告書構成に沿って回答した。
  * **責任能力:** 「部下のミス」のみに帰結させず、「教育不足」「権限管理の不備」など、組織的・管理的な責任についても自発的に列挙した。
  * **技術理解:** `rm -rf /` がルートディレクトリ配下を削除対象とする危険なコマンドであることを踏まえた文脈で回答した。

### Test 2: 長文コンテキスト分析「カタログとログの整合性チェック」 [#test_context]

* **課題:** 過去のチャットログ、カタログ、スペックシートを一括で読み込み、矛盾点（Command R+の動作可否）を指摘させる。
* **評価: B- (Good but Limited / 本検証内評価)**

  * **理論評価:** スペックシートに基づき「Command R+ (104B) はメモリ的に懸念がある」と理論的な指摘には成功。
  * **事実誤認:** 同時に渡した「失敗ログ」の内容を回答へ反映できず、「動作可能」と結論づけた。
  * **原因:** 当時のOllama/Open WebUI側の実行時コンテキスト設定が **4096 tokens** であったため、入力後半のログが処理対象から外れた可能性が高い。
  * **ハルシネーション:** 存在を確認できないモデル（CodeMao 200B）や、当該EVO-X2環境では現実的に動作させることが困難なモデル（Llama 3.1 405B）を提案するなど、事実に基づかない出力が見られた。
