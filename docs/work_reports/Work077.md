# Work077: Gemini WebUIにおける1013エラーの調査とプロンプト誤検知の回避

Tag: [[Gemini]], [[Troubleshooting]], [[Error1013]], [[PromptInjection]], [[DevTools]]

## 1. 目的

- Geminiブラウザ版でプロンプト送信時に発生する原因不明のエラー「1013」の原因を調査する
- ネットワーク、ブラウザ拡張機能、キャッシュ、入力内容などの各レイヤーから要因を切り分ける
- 将来的な同種エラー発生時のトラブルシューティング手順と、有効な回避策をナレッジとして確立する

## 2. システム環境

- **Node:** クライアントPC (Windows / Google Chrome 147.0.7727.138)
- **Target:** Gemini Web版 (APIエンドポイント: `/chat` または `StreamGenerate`)
- **Method:** ブラウザからの直接プロンプト入力、DevToolsを用いた通信状況・リクエスト内容の確認

## 3. 作業ログ・解決プロセス

### 発生した課題

- 通常のプロンプトを送信すると画面上に「エラー（1013）」と表示され、AIからの応答が一切返ってこない
- 同じブラウザ内の別タブでは正常に動作し、特定のタブでのみエラーが再現する状況だった
- DevToolsのNetworkタブを確認したところ、Google Analyticsのトラッキング通信 (`/g/collect`) が `ERR_ADDRESS_INVALID` で落ちているのが見つかったが、これは広告ブロッカー等による遮断であり、1013エラーの直接の原因ではないダミー（Red Herring）と判断した
- シークレットウィンドウで実行すると正常に動作したため、ブラウザ拡張機能を疑い全てOFFにしたが、通常ウィンドウでは依然として1013エラーが発生し続けた

### 解決策

- DevToolsの解析を深堀りした結果、Geminiの中核機能である `StreamGenerate` の通信自体は **HTTP Status 200 (OK)** で成功していることを確認した
- これにより、単純な通信プロトコル層（WebSocket切断等）のエラーではなく、Gemini側でリクエスト内容または応答生成過程を処理した結果として返される「アプリケーション層の内部エラー」である可能性が高いと判断した
- 最終的な原因として強く疑われたのは、入力したプロンプト内の文言（「AIに指示を出す」「前提条件を明確にする」「いきなり作業しないで」「作業の手を止めさせて」など）だった
- これらの「人間に対する仕事の振り方（指示）」のロジックが、AIの基本動作を強制停止させたり制限を上書きしようとする**プロンプトインジェクションのコマンド**に類似していたため、Gemini側の入力検査または安全制御により、危険な通信として誤検知された可能性がある
- 対象の文章全体を **コードブロック（\`\`\`）** で囲んで送信することで、システム側が「AIへの直接の命令」ではなく「引用テキスト（データ）」として扱いやすくなり、当該ケースでは送信可能になった

## 4. 成果物

### エラーを回避できたプロンプトの記述方法（コードブロックによるエスケープ処理）

````text
あー。プロンプト全体を「```」で囲ったら通った。
プロンプトそのものに問題があったのか・・・

```
これ、私がAIに指示を出すときに強く意識してることです。
①前提条件を明確にする
②ゴール設定を明確にする
③いきなり作業しないで、先に方針を出させる
④出来ない場合は、作業の手を止めさせて、その理由を列挙させる

AIに指示を出すつもりで、人間相手にも同じロジックで指示を出したんですが、円滑に作業がスタートしました。
私はこれを「AI活用術」だと思ってましたが、そもそも仕事の頼み方の正しい方法だと感じました。
```
````

### 最終結論

* Gemini WebUI上で発生する「1013エラー」は、通信障害だけでなく、入力内容に対する安全制御またはプロンプトインジェクション検知に近い仕組みによって発生する可能性がある。
* ユーザー側に悪意がなくても、システムへの強い命令形（停止、前提の上書き、作業制御など）に類似した文章は、誤検知の対象になり得る。
* 検証テキストや指示書の草案など、AIへの直接の命令として解釈されたくない文字列を含ませる場合は、プレーンテキストのまま送らずに「```」でラップして引用テキストとして扱わせる運用が有効である。
