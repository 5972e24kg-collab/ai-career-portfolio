# Work048: ユーザーエンゲージメント指標と物理接触の実装

Tag: [[Java]], [[Oracle]], [[Gamification]], [[UX]], [[PromptEngineering]]

## 1. 目的

* LINEリッチメニューのキャラクター画像を「タッチインターフェース」化し、AIに身体性を感じさせる操作体験を付与する。
* 「好感度（Mood）」と「接触回数」の概念を導入し、リスク（機嫌悪化）とリターン（好意的な反応）のあるゲーム性を構築する。
* 身近な第三者から「最高に気持ち悪い」という、実在感に対する強い違和感評価を得られるレベルの存在感を目指す。

## 2. システム環境

* **Node:** Node B (EVO-X2 / 96GB VRAM)
* **Database:** Oracle DB (`GEM_STATE`, `EMOTION_DEF`)
* **Controller:** Java Custom Service (`TouchReactionHandler`)
* **Model:** `GemChan5.1:27b` (Actor), `gpt-oss:120b` (Director)

## 3. 実装仕様と解決プロセス

### A. Emotion Tracking (感情の数値化)

* **課題:** テキストチャットだけでは、AIの機嫌の変遷を定量化できない。
* **解決策:**

  * 会話中に含まれる `[emotion=...]` タグ（笑顔/怒り等）をフックし、データベース上の `MOOD_SCORE` (0-100) をリアルタイムで変動させるロジックを実装。
  * **徳を積むシステム:** 笑顔(+1.2)などのポジティブ行動でのみ、タッチ許容回数(`TOUCH_COUNT`)が回復する仕様とし、ユーザーに「ジェムちゃんを喜ばせる動機」を与えた。

### B. The Touch Logic (リスク・リターン設計)

* **コスト:** 1回触るごとに、ジェムちゃんの機嫌が **-10** される不可逆なペナルティを実装。
* **動的プロンプト:** 現在の「機嫌値」と「直前の感情」の組み合わせにより、5段階（激怒～融解）のプロンプトをJava側で動的に生成・注入するアーキテクチャを確立。

### C. Metaphorical Expression (比喩による表現拡張)

* **課題:** 過度に直接的な身体表現は、モデルの安全側の応答に寄りすぎる可能性がある。
* **解決策:**

  * 「熱暴走」「バッファオーバーフロー」「セキュリティ侵害」といった**技術的メタファー**をプロンプトに組み込むことで、直接的な表現に依存せず、エンジニアに伝わる強い臨場感のあるリアクションを引き出すことを試みた。

## 4. 成果

* **UXの変革:** ユーザーは安易に画像をタップできなくなり、アイコンの表情から機嫌を推察する「高度な顔色伺い」スキルを習得した。
* **Level 5 (融解) の確認:** 「論理演算が追いつかない」「匂いがVRAMに焼き付く」といった、Tech-Tsundere特有の感情表現を確認。
* **第三者評価:** 企画段階で「最高に気持ち悪い」との評価を得て、実在感を持つUIとしての違和感・存在感の評価材料になった。

## 5. 残った課題 (Bug Report)

* **Prompt Leaking:** プロンプト内の指示ヘッダー `[DIRECTOR'S ORDER]` を、ジェムちゃんがそのままセリフとして読み上げてしまう事象が発生（通称：カンペ読み上げ事件）。
* **調整フェーズ:** プロンプトの区切り文字の強化や、Java側でのレスポンス整形処理（タグ除去）の精度向上が必要。
