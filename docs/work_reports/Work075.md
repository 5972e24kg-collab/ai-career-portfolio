# Work075: Codexバイブコーディング環境構築とPTAクイズゲームUI改造検証

Tag: [[Codex]], [[VibeCoding]], [[React]], [[VSCode]], [[Windows11]], [[EnvironmentSetup]], [[Troubleshooting]]

## 1. 目的

* Codexを用いた「バイブコーディング」環境を、専用のWindows端末上に安全に構築する。
* メイン開発機・本番環境・関係者向け本番サーバーから隔離した、壊れてもよいAIコーディング検証環境を用意する。
* 既存のReact製4択クイズアプリを題材に、CodexへUI改造を自律実装させる体験を行う。
* 「AIの手綱を強く握る」従来型の使い方ではなく、「仕様と制約だけ与えて、演出・デザインはAIに任せる」開発スタイルを体験する。
* 追加PCでも再現できるように、環境構築手順を完全な手順書として資産化する。
* 最終成果物として、`yarn build` 可能なReactアプリを得る。
* 本番環境へのデプロイは人間が行い、Codex端末には本番権限を持たせない。

## 3. 作業ログ・解決プロセス

### 発生した課題

#### 課題A: バイブコーディング用の安全な隔離環境が必要だった

メイン開発機でCodexを自由に動かすことにはリスクがあった。

* 既存の重要開発環境を壊す可能性がある
* 本番用認証情報やSSH鍵に触れる可能性がある
* 関係者向け本番環境や自宅AI基盤へ誤接続する可能性がある
* 今回の目的は「AIに自由に作らせる体験」であり、通常の厳密な制御型開発とは性質が異なる

#### 解決策

専用のN100ミニPCを用意した。

* N100 / 16GB / Windows 11 Pro
* 誰も使っていないPC
* 壊れてもよい
* 本番環境への認証情報を置かない
* Gitはローカルリポジトリのみ
* 本番デプロイは人間が担当
* Codexには `yarn start` / `yarn build` / ファイル編集のみを担わせる

#### 課題D: PowerShell実行ポリシーにより `npm.ps1` がブロックされた

追加で用意した端末にて、Node.js導入後に `npm -v` を実行したところ、以下のエラーが発生した。

```text
npm : このシステムではスクリプトの実行が無効になっているため、ファイル [Path masked]\npm.ps1 を読み込むことができません。
```

#### 解決策

PowerShellの実行ポリシーを `RemoteSigned` に変更した。

推奨はユーザー単位の変更である。

```powershell
Set-ExecutionPolicy -Scope CurrentUser RemoteSigned
```

今回の検証では以下でも解消した。

```powershell
Set-ExecutionPolicy RemoteSigned
```

確認が表示されたら `A` または `Y` を入力する。

```text
[Y] はい  [A] すべて続行  [N] いいえ
```

その後、以下で確認した。

```powershell
npm -v
```

`yarn install` 実行時に、以下のような警告が出た。

```text

```

#### 解決策

```powershell

```

ただし、既存プロジェクトがnpm運用の場合は、逆に `yarn.lock` を使わず `npm install` / `npm start` / `npm run build` に統一する。

#### 課題H: Codex実行時に `VCRUNTIME140.dll` が見つからない端末があった

追加で構築した端末の一部で、Codex実行時に以下のシステムエラーが表示された。

```text
VCRUNTIME140.dll が見つからないため、コードの実行を続行できません。
プログラムを再インストールすると、この問題が解決する可能性があります。
```

#### 原因

Visual C++ Runtime が不足していた。

#### 解決策

Microsoft Visual C++ Redistributable x64 を導入した。

PowerShellで以下を実行する。

```powershell
$work = "$env:TEMP\vc_redist"
New-Item -ItemType Directory -Force -Path $work | Out-Null

Invoke-WebRequest "[https://aka.ms/vc14/vc_redist.x64.exe](https://aka.ms/vc14/vc_redist.x64.exe)" -OutFile "$work\vc_redist.x64.exe"
Start-Process "$work\vc_redist.x64.exe" -ArgumentList "/install /passive /norestart" -Wait
```

実行後、Codexを再実行する。

---

#### 課題I: ESLint warning が残った

`yarn start` / `yarn build` 時に以下の警告が表示された。

```text
React Hook useEffect has a missing dependency: 'loadQuestions'. Either include it or remove the dependency array
```

#### 判断

これは既存コード由来の警告であり、ビルド失敗ではない。
今回の主目的であるCodex環境構築・UI改造・`yarn build` 生成には支障がない。

#### 今後の対応

Codexに以下のように依頼して修正可能。

```text
useQuizEngine.js の React Hook useEffect の ESLint warning を解消してください。
ただし、クイズの挙動は変えないでください。
yarn build が通ることを確認してください。
```

---

#### 課題J: Codexが極めて短時間でUI改造を完了した

当初は半日程度かかる可能性を想定していたが、Codexは数分程度で以下を実施した。

* 既存コード分析
* 改造方針提案
* ゲーム風UI実装
* スタート画面追加
* イベント設定ファイル追加
* クリアキーワード表示
* CSSアニメーション追加
* `yarn build` 実行

#### 分析

成功要因は以下。

* 元アプリの構造が人間にもAIにも読みやすかった
* `App.js` / `useQuizEngine.js` / `components` / `engine` の責務分離が明確だった
* 「壊してはいけない部分」と「自由にしてよい部分」を明確に指示した
* 「小学生向け」「大魔王の挑戦状」というAIが解釈しやすいテーマを与えた
* 既存アプリ自体が、AIにとって実質的な仕様書として機能した

# 6. 学習結果・考察

## 6.1 今回うまくいった理由

### 既存アプリがAIにとって良い仕様書だった

元のReactアプリは責務分離が明確だった。

```text
App.js
  画面分岐

useQuizEngine.js
  クイズロジック

quizCsvParser.js
  CSV読み込み

components/
  画面表示
```

この構造により、Codexは「どこを触るべきか」「どこを触らないべきか」を判断しやすかった。

---

### 制約と自由度の分離が適切だった

今回の指示では以下を明確に分けた。

```text
壊してはいけない:
- quiz.csv読み込み
- CSV形式
- 3問正解クリア
- 提供元表示
- 本番デプロイ禁止

自由にしてよい:
- デザイン
- 演出
- 文言
- CSS
- スタート画面
- ゲージ
- 魔王城
- 宝箱
```

この分離が、Codex의 自律実装を成功させた。

---

### テーマ指定がAIにとって解釈しやすかった

単に「いい感じにして」ではなく、以下のように文脈を与えた。

```text
- 小学生向け
- 関係者向けイベント用
- QRコードからスマホで遊ぶ
- 大魔王の挑戦状
- 3問正解でキーワードGET
```

このため、Codexは「ゲーム風UI」の方向性を具体的に解釈できた。

---

### AIに任せる領域としてUI/演出が適切だった

ロジックの正確性を求める領域ではなく、既存ロジックを維持した上でのUI改造だったため、Codexの創造性が活きた。

特にCSSは人間側では査読が難しい一方、実機で見た目を確認できる。
そのため、コードレビューではなくユーザーテストで品質確認する対象として割り切ることができた。
