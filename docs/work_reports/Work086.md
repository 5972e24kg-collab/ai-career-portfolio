# Work086: AIレビューを用いたHITL型ポートフォリオ改修と実証記録

Tag: [[HITL]], [[AIレビュー]], [[技術ポートフォリオ]], [[LLM]], [[Claude]], [[ChatGPT]], [[Human-in-the-loop]], [[AI駆動開発]], [[FailureAnalysis]], [[Portfolio]]

## 1. 目的

* 技術ポートフォリオ記事「記憶と文脈を維持するAIエージェント」を、外部AIレビューの指摘をもとに再構成する。
* Claudeによる外部レビューを壁打ち材料として使い、記事の不足点・弱点・説得力不足を抽出する。
* ChatGPTを用いて、レビュー内容・既存Work・本人の補足説明を統合し、HTML記事へ反映する。
* 人間が方針判断・情報補足・採否判断を行い、AIが構成案作成・文章化・HTML化を支援するHITL型の編集プロセスを実践する。
* 「AIの補助なしでは不可能だが、AIへの丸投げでも不可能」という、AI活用型開発・編集の実態を記録する。
* 完成品だけでなく、レビュー、判断、素材収集、再構成、検収、公開までのプロセス自体を、技術ポートフォリオの実践記録として残す。

## 3. 作業ログ・解決プロセス

### 発生した課題

#### 3.1 初稿は完成度が高い一方で、失敗と実証の記録が薄かった

初稿の「記憶と文脈を維持するAIエージェント」は、以下の内容を整理できていた。

* LINEを入口にしたローカルLLMエージェント
* Oracleによる会話履歴・記憶管理
* Director / Actor分離
* 短期記憶・長期記憶
* 音声合成
* React / VRMアバター
* 外部デモ用スクリーンショット枠

しかし、Claudeレビューでは、記事が「完成したアーキテクチャの説明」としては良い一方で、以下が不足していると指摘された。

* 本当に動いた証拠
* 外部LTで使った動画
* Work082 / Work083 / Work085に記録された実地運用の失敗
* 失敗から現在の設計に収束した経緯
* Actorモデル変遷の時系列説明
* 「なぜこの構成になったのか」の泥臭い根拠

この指摘により、記事の方向性を以下へ変更した。

```text
完成形の紹介
↓
失敗を踏み抜いた結果、この設計に収束したことを示す実証記事
```

#### 3.2 動画・スクリーンショット掲載に対する心理的抵抗があった

外部LTで使用した動画は、アバター・音声・キャラクター性が強く出ているため、技術ポートフォリオに掲載するとエンターテイメント寄りに見える懸念があった。

一方で、動画は以下を一度に示せる強い証跡だった。

* LINE入力
* Director / Actor構成の説明
* 音声出力
* VRMアバター表示
* 外部LTで実際に使ったデモであること

検討の結果、動画を「エンタメ要素」ではなく「動作証跡」として扱う方針にした。

```text
動画をトップで大きく見せる
  → 採用しない

本文中に小さく、実演デモとして掲載する
  → 採用
```

#### 3.3 Actorモデルの表記が、Work間で時系列的に揺れていた

Claudeレビューでは、Actorモデルの表記に以下の揺れがあると指摘された。

```text
Work064 / Work065 / Work067:
  Gemma 3:27b

Work071:
  gemma4:e4b / GemChan6.x:e4b

Work072:
  GemChan7.1:26b

概要記事:
  gemma4:26b(FT)
```

これは矛盾ではなく、時系列上のモデル差し替えである。

実際の流れは以下。

```text
Framework v1:
  gemma3:27b(FT)
  または gpt-oss:120b との自動切り替え

Framework v2初期:
  gemma4:e4b(FT)

Framework v2安定化:
  gemma3:27b(FT) へ戻す

Framework v2現在:
  gemma4:26b(FT) / GemChan7.1:26b
```

記事では、この表記揺れを隠すのではなく、Actorを差し替えながら運用検証した履歴として明示する方針にした。

#### 3.4 gemma4:e4bのXML崩れログは残っていなかった

v2化後、`gemma4:e4b(FT)` は短期間Actorとして実戦投入された。

しかし、以下の理由により、当時の具体的なXML崩れログは残っていなかった。

* フレームワークv2切り替え: 2026年4月17日
* gemma4:26b QLoRA再挑戦: 2026年4月20日
* e4bからgemma3:27bへ戻したのは、おそらく4月17日〜20日の間
* データベース上のLLM全文ログは、2026年5月31日時点で5月3日以降のみ保持
* DBの全文ログ保持期間は4週間
* Actorの出力ログは成功ログのみで、XML崩れログは残っていない

このため、記事では以下の方針を採用した。

```text
具体的なXML崩れ例は掲載しない。
ログが残っていないことを明記する。
時系列上、短期間でe4bを退避した事実を記録する。
```

これは、証拠のない失敗例を捏造しないための判断である。

#### 3.5 長期記憶は、最初から成功したわけではなかった

Work050では、1日分の会話ログを `gpt-oss:120b` で主観的な日記へ圧縮し、長期記憶として翌日以降へ引き継ぐ仕組みを作った。

しかしWork051では、その日記を `role=assistant` として履歴に挿入した結果、以下の問題が発生した。

```text
日記文字数:
  約600文字

平均応答文字数:
  166文字 → 422文字へ増加

発生した問題:
  長文化
  語彙の不自然化
  キャラクター崩壊
  システムプロンプトの150文字制限を無視
```

この失敗は、記事における重要な根拠になった。

単に「記憶を追加したら賢くなった」のではなく、

```text
記憶を追加したら壊れた。
だから記憶の渡し方を設計した。
```

という流れを明示できるためである。

#### 3.6 gpt-oss:120b常駐には、ROCm運用上の障害があった

Work062では、`gpt-oss:120b` をROCm環境に常駐させた際、推論していないにもかかわらずCPUファンが暴走する問題が発生していた。

観測された事象は以下。

```text
ACPI温度センサー:
  97℃

top -H:
  Ollamaの特定スレッドがCPU 1コア分を常時占有

原因:
  ROCm HSAランタイムのビジーウェイト
```

解決策は以下。

```yaml
environment:
  - HSA_ENABLE_INTERRUPT=1
  - HIP_VISIBLE_DEVICES=0
```

この解決は、単なる温度トラブル対応ではなかった。

これにより、

* Framework v1での `gemma3:27b` / `gpt-oss:120b` 自動切り替え
* Framework v2での `gpt-oss:120b` Director常駐

が現実的になった。

そのため、記事ではローカルLLM常時運用の重要な前提として追記した。

#### 3.7 VRMアバターは、表示できれば終わりではなかった

Work064 / Work065および本人補足から、VRMアバター側にも多くの失敗があった。

主な課題は以下。

```text
SpringBone重力設定によるモデル消失
R3Fのclock.getDelta()による物理演算停止
カメラPositionだけでは足元表示が解決しない問題
Mixamo FBXとVRMのボーン軸不一致
Zustandの再レンダリング地獄
モーションごとの初期位置・角度の差異
ワンショットアニメーション後の硬直
字幕吹き出しの追従問題
```

さらに、「自然に見える振る舞い」を作るために、単なるランダムモーションではなく、以下の2層状態管理を導入していた。

```text
コアステータス:
  余剰活力
  眠気
  退屈度
  注目度
  機嫌値
  甘え / デレ重み

派生値:
  眠そう度
  動きたさ
  接近欲
  注視ターゲット
  表情温度
  話したさ
```

LLMやIoT、時間経過はコアステータスに影響し、実際のアバター行動は派生値から決定する。

これにより、LLM出力をそのままアバターへ流すのではなく、状態管理レイヤを挟んで身体性を演出していることを記事へ追加した。

#### 3.8 外部デモで、音声だけ届かない問題が露出した

Work083では、外部デモ環境に持ち出したことで、初めて以下の課題が見えた。

```text
React画面:
  外部から見える

SSEによる字幕・表情:
  外部から動作確認できる

音声:
  ローカル再生環境に依存しており、外部デモでは届かない
```

これに対して、以下の設計へ変更した。

```text
SSE:
  何が起きたかを通知する制御プレーン

HTTP / nginx:
  WAVファイルを配信するデータプレーン
```

実装上のポイントは以下。

```text
1. Java WebAPIがStyleBertVITS2で音声合成
2. [voice-public-path]/[ChatNo]-[SEQNo].wav.tmp へ保存
3. 書き込み完了後に .wav へrename
4. nginxでHTTP公開
5. SSEでaudioPathだけをReactへ送信
6. React側で音声再生キューへ投入
```

この失敗談は、「外に出すことで初めて要件が露出する」例として記事へ追加した。

#### 3.9 LT本番では、ライブ音声・SSE・デモ環境運用の課題が露出した

Work085では、AI交流会LT本番で以下の課題が露出していた。

```text
ライブデモでアバター音声が会場に出なかった
Google Meet / 会場音響 / PCスピーカー / Bluetoothイヤホンの経路設計不足
ログ表示用SSE streamが動いていない場面があった
SSE状態を画面から確認できなかった
デモ環境と本番環境の切り替えがコメントアウト運用だった
観客には内部構成が十分伝わり切らなかった可能性があった
```

これらは失敗ではあるが、技術記事においては非常に価値が高い。

理由は、実際に外部発表しなければ得られない運用課題だからである。

記事では、これを「LT本番で露出したライブデモ運用の課題」として追加した。

#### 3.10 Claudeレビューから、HITLそのものが実演されていると指摘された

Claudeは、今回の改修プロセスそのものを以下のように評価した。

```text
AIの補助なしでは不可能。
しかしAIへの丸投げでも不可能。
判断は人間、製造はAI。
```

これは、Work079で扱った「バイブコーディングではないAI開発」の方法論と同じ構造である。

今回の流れは以下。

```text
1. 人間が初稿を作る
2. Claudeが外部レビューする
3. 人間がレビューの妥当性を判断する
4. ChatGPTが方針整理とHTML化を支援する
5. 人間が不足情報を補足する
6. ChatGPTが再構成する
7. 人間が検収する
8. 公開する
```

つまり、記事の内容だけでなく、記事を改善する過程そのものがHITLの実践になっていた。

### 解決策

#### 3.11 追加する失敗談を、記事主題に直結するものだけに絞った

失敗談は大量に存在するが、すべて入れると記事が散漫になる。

そのため、以下の基準で採用した。

```text
採用条件:
  現在のアーキテクチャに直接つながる失敗であること
  記憶と文脈を維持するAIエージェントという主題に関係すること
  外部公開や実運用の説得力を高めること
  既存Workまたは本人補足で説明可能であること

不採用または抑制:
  MCP / Codex系など、別代表記事の主題に近いもの
  面白いが今回の記事の焦点を散らすもの
  証拠がなく再現不能な失敗例
```

採用した失敗談は以下。

```text
Actorの語彙崩壊・演技ブレ
gemma4:e4bの構造化出力不安
gemma4:26b QLoRAのリベンジ成功
長期記憶の日記挿入によるContext Infection
ROCmスピンロックによるCPUファン暴走
VRM / R3F / Zustand / モーション制御の失敗
外部デモでの音声欠落
LT本番での音声経路・SSE・環境分離の課題
```

#### 3.12 記事構成を「完成形」から「失敗からの収束」に再設計した

既存記事の基本構成は維持しつつ、以下の章を追加・強化した。

```text
6-1. Actorモデルの変遷と差し替え可能性の検証
7-1. 記憶を増やせばよいわけではなかった
9-4. 実演デモ動画
10-1. 外部デモで初めて露出した音声配信の欠落
10-2. 身体性も、LLM出力をそのまま動かすだけでは成立しなかった
11. 外部公開と実証：部屋の中のAIを外に出して分かったこと
12-1. gpt-oss:120b常駐とROCmスピンロック
13. 失敗から得た設計原則
```

これにより、記事の重心を以下へ変更した。

```text
完成したアーキテクチャの説明
↓
失敗を踏み抜いた結果、現在の設計に収束した記録
```

#### 3.13 Actor変遷は、矛盾ではなく設計価値として扱った

Actorの世代表記の揺れは、記事内で以下のように整理した。

```text
Framework v1
├─ Actor: gemma3:27b(FT)
└─ 補助的に gpt-oss:120b へ自動切り替え

Framework v2
├─ Actor: gemma4:e4b(FT)
│  └─ 新世代モデルとして期待したが、実戦では構造化出力の安定性に不安が残った
├─ Actor: gemma3:27b(FT)
│  └─ 安定性が改善。長期運用には20B級以上のモデル規模が必要と判断
└─ Actor: gemma4:26b(FT)
   └─ QLoRA成功後に本番Actorとして移行
```

また、e4bの失敗ログが残っていないことも明記した。

```text
具体的なXML崩れの実例は掲載しない。
当時の失敗レスポンス全文はログ保持期間外のため残っていない。
```

そのうえで、現在の安定運用の数字を掲載した。

```text
2026年5月30日 22:00:08時点
v2化後の会話ログ No.573
うち約400件を26B Actorが担当
この範囲ではXMLフォーマット崩れを観測していない
```

なお、ここで示す日時・ログ保持期間・会話ログ件数は、検証範囲を示すためのメタ情報である。会話本文、個人情報、接続情報、DBスキーマ、内部URL、ホスト名、IPアドレスは公開対象に含めない。

#### 3.14 失敗から現在の設計への対応表を追加した

記事後半に「失敗から得た設計原則」として、以下の対応表を追加した。

```text
踏み抜いた失敗
見えた原因
現在の設計
```

代表例。

```text
Actorに文脈理解と演技を両方背負わせた
→ コンテキスト蓄積で語彙が偏り、演技品質もターンごとに変動した
→ Director / Actor分離

日記形式の長期記憶をrole=assistantへ挿入した
→ In-context Learningにより、長文日記の文量・文体が感染した
→ Long-term Memoryの扱いを再設計、Dual-Phase Thinkingを経てDirectorが記憶を台本化

外部デモで声だけ届かなかった
→ 音声再生がローカル環境に依存していた
→ SSEを制御プレーン、HTTP/nginxを音声データプレーンとして分離
```

これにより、失敗談が単なるエピソードではなく、設計判断の根拠として機能するようになった。

```text
  Actor変遷の正確な時系列説明
  ログが残っていない事実の明示
  記事に入れるべき失敗談の判断
  最終検収

ChatGPT:
  方針整理
  記事構成の再設計
  失敗談と設計判断の対応付け
  HTML生成
  技術レポート化
```

この流れは、AIに丸投げしたものではない。

AIは大量の文章生成と構成整理を高速化したが、以下は人間が担当した。

```text
何を載せるか
何を載せないか
どこまで公開するか
証拠がないものをどう扱うか
心理的抵抗のある動画を載せるか
技術記事としての重心をどこに置くか
```

### 4.4 HITLとしての実証ポイント

今回の作業は、HITLの実例として以下を示している。

```text
1. AIレビューは、人間が受け止めて判断しなければ価値にならない。

2. AI生成は、人間が素材・方針・制約・公開範囲を与えなければ、正しい成果物にならない。

3. 人間だけで同等の再構成・HTML化・整合確認を短時間で行うことは難しかった。

4. AIだけでは、どの失敗談を載せるべきか、どこまで失敗や心理的抵抗のある要素を出すべきか、何を公開リスクと見るかを判断できない。

5. 人間の判断軸とAIの製造能力が噛み合った時、短時間で公開品質の成果物に到達できる。
```

### 4.5 今回の作業を一文で表すなら

```text
Claudeのレビューを壁打ち材料にし、人間が公開判断と技術的補足を行い、ChatGPTで記事構成とHTMLを再製造し、失敗を設計原則へ変換したHITL型ポートフォリオ改修作業。
```

### 4.6 最終結論

今回の作業は、単なる記事修正ではなかった。

もともと記事の主題は、記憶と文脈を維持するAIエージェントだった。

しかし、Claudeレビュー、ChatGPTによる構成支援、人間による判断・補足・検収を経て、記事自体が次の段階へ進んだ。

```text
完成した成果物を見せる記事
↓
失敗から設計に収束した過程を見せる記事
```

さらに、その改修プロセス自体が、HITLの実演になっていた。

```text
AIの補助なしでは不可能。
しかし、AIへの丸投げでも不可能。
判断は人間、製造はAI。
```

これは、過去のWork079で扱った「バイブコーディングではないAI開発」と同じ構造である。

今回、技術ポートフォリオの記事を評価・修正・公開する行為そのものが、HITL型AI活用の実践例になった。

その意味で、今回の成果物はHTML記事だけではない。

最大の成果物は、以下である。

```text
AIレビューを受け、人間が判断し、AIで製造し、人間が検収することで、
公開品質の技術ポートフォリオへ短時間で到達できることを実証したプロセス。
```
