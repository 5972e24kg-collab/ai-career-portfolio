# Work085: AI交流会LT完遂とチャットボット外部発表による視点・課題・経験の獲得

Tag: [[LT]], [[AI交流会]], [[Gemchan]], [[LINE]], [[LLM]], [[AIエージェント]], [[Demo]], [[Presentation]], [[Human-in-the-loop]], [[振り返り]]

## 1. 目的

* LINE Developer Community / AI交流会のLT枠で、ジェムちゃんを外部向けに紹介する
* LINEを入口にしたローカルLLMエージェントとして、ジェムちゃんの全体像・AI/LLM構成・デモ画面・活用実績を説明する
* 初めてのLT登壇を通じて、人前で自作AIエージェントを発表する経験を得る
* 観客がジェムちゃんのどこに反応し、何を疑問に思うのかを実地で観測する
* 事前に用意した台本、動画、静止画、ライブデモ、懇親会デモという複数のリスクヘッジが有効だったかを検証する
* 外部会場でのGoogle Meet / 音声 / アバター / SSE / VPN / デモ環境運用上の課題を洗い出す
* 今後のLT・面接・技術ポートフォリオに向けて、ジェムちゃんの見せ方と説明方針を再設計する材料を得る

## 3. 作業ログ・解決プロセス

### 発生した課題

#### 3.1 緊張により、計画通りの発表進行が難しかった

初めてのLT登壇であり、発表者自身の緊張が非常に大きかった。

事前には、印刷した台本を片手に読み上げるだけの想定だったが、実際には緊張により、台本から脱線したり、逆に端折ったりした。

発表後、何を話したのかはあまり記憶に残っていない。

ただし、以下の説明骨格はおそらく伝えられた。

* ジェムちゃんはLINEを入口にしたローカルLLMエージェントである
* Director / Actor という2つのAIエージェントが協力して動作している
* Directorは監督、Actorは女優という役割で説明できる
* LINE返信、音声、アバター、記憶生成が連動している
* 埼玉スタジアムのイベントで外部実地稼働した

今回、発表内容をエンターテイメント方向に振ったが、それが逆に緊張を高めた可能性がある。

自分自身の性質としては、趣味の成果を自分のロールで発表するよりも、システム構成を解説員として説明する方が向いていると感じた。

#### 3.2 チャットボットの仕組み自体は、観客に十分伝わり切らなかった可能性がある

発表後の懇談会では、内部構成や仕組みに関する質問が多かった。

主な質問は以下。

* どうやって動いているのか
* どういう仕組みなのか
* ローカルLLMとは何か
* なぜDirector / Actorに分けているのか
* LINEから自宅LLMへどう接続しているのか

自分にとっては「ローカルでLLMを動かす」ことは日常である。

しかし、会場の参加者から見ると、Gemini / ChatGPT / Claude などのクラウドAIやAPI利用は分かっていても、自宅で大規模LLMを動かし、LINEと連携し、音声とアバターまで接続している構成はかなり異質だった。

特に、Director / Actor 構成は概念自体が十分浸透していなかった。

自分は過去に、Gemma3:27bへ以下をすべて詰め込もうとして失敗した経験がある。

* 文脈理解
* 語彙の維持
* キャラクター演出
* 応答制御
* 記憶の扱い

その失敗から、判断するAIと演じるAIを分離する必要性を感覚的に理解していた。

一方で、フロンティアモデルやクラウドモデルを使う場合、多くの人は1回のプロンプトで命令を完遂させる発想になりやすい。

そのため、「なぜ役割を分ける必要があるのか」は、こちらが思っている以上に説明が必要だった。

#### 3.3 アバターと音声のインパクトが非常に大きかった

今回、都内高層階のイベント会場という、非常に非日常感のある会場で発表した。

ガラス張りの高層階の夜景を背に、巨大スクリーンにジェムちゃんのアバター画面が映し出された。

さらに、事前録画した動画では、ジェムちゃんがダンスを踊り、大音量で以下のようなツンデレ音声が流れた。

```text
私はジェムよ。
DirectorとActorの二段構成で動いてるの。
構成は理解できた？
```

この映像・音声のインパクトは非常に大きかった。

発表者自身も強く動揺するほどのインパクトがあり、観客にも一定の印象を残した可能性がある。

一方で、アバターと音声の印象が強すぎるため、内部アーキテクチャの説明が相対的に埋もれた可能性がある。

次回は、見た目のインパクトを入口にしつつ、より明確に内部構成へ誘導する必要がある。

#### 3.4 ライブデモで音声が再生されなかった

ライブデモでは、アバター音声が会場に再生されなかった。

原因は、Google Meetの画面共有でスライドを発表するという運用を十分に想定できていなかったことだった。

当初想定していた音声経路は、ノートパソコンのスピーカーからアバター音声を再生する形だった。

しかし、Google Meet接続中にノートパソコンのスピーカーから音声を出すと、マイクが音を拾い、ハウリングする可能性があった。

ミュートすると、今度はアバター音声が再生されない。

当日、発表30分前にこの問題に気が付いた。

勉強会がすでに始まっており、時間が限られている中で、以下を試した。

* ノートパソコンにBluetoothイヤホンを接続
* Google Meetの音声をイヤホンへ送る
* アバターの音声をパソコンスピーカーへ出す
* ハウリングを防ぎつつ、アバター音声だけを会場に出す

しかし、結果として失敗した。

後の検証で、アバター音声もBluetoothイヤホン側へ吸われていたことが分かった。

音声経路の分離をした状態での動作確認が不足していた。

今後は、Google Meet / 会場マイク / PCスピーカー / Bluetoothイヤホン / アバター音声 / スライド動画音声を含めた音声経路設計を、事前にテストする必要がある。

#### 3.5 SSEのstreamが動いていない箇所があった

アバター画面には、SSEが2系統存在する。

* アバター制御用stream
* ログ表示用stream

このうち、ログ表示用streamが動いていない時があった。

短時間のデモ動作確認で、複数の機材から接続テストを繰り返したことが影響している可能性がある。

streamを担当しているDockerコンテナを再起動すれば解消するが、アバター画面からはそれが確認できない。

そのため、異常検知と対処が後手に回った。

現在のアバター画面には「音声再生」の動作確認ボタンがある。

同じように、SSEの動作確認ボタン、接続状態表示、最終受信時刻表示が必要である。

#### 3.6 Reactアバター画面の本番環境とデモ環境の分離が不十分だった

現在、Reactアバター画面は以下のブランチで分離している。

* `main`: 自宅本番用
* `wip/demo`: デモ環境用

主な分離対象は以下。

* WebAPIの接続先
* アバター衣装
* 内部表示項目
* デモ用の画面構成

現状では、ソースコード内に記載されている接続先情報や、衣装カタログをコメントアウトで切り替えて対応した。

今回は乗り切れたが、今後デモ機会が増える場合、この方式では管理が難しい。

デモ環境と本番環境の差分は、ブランチ差分ではなく設定ファイルや環境変数で切り替えるべきである。

例として、以下のような設定が必要になる。

```text
APP_MODE=home | demo
API_BASE_URL=...
SSE_AVATAR_URL=...
SSE_LOG_URL=...
AVATAR_COSTUME=formal
ENABLE_COSTUME_ROTATION=false
SHOW_DEBUG=false
ENABLE_LOCAL_IP=false
```

これは単なる便利化ではなく、デモ運用における再現性と安全性の確保である。

#### 3.7 自分が思うアピールポイントと、他者が注目するポイントに差があった

自分にとって、ジェムちゃんの強みは以下だった。

* 長期記憶と短期記憶
* Director / Actor構成
* 自作フレームワーク
* ローカルLLM
* QLoRAファインチューニング
* 本番環境とデモ環境の分離
* LINE / 音声 / アバター / 記憶生成の統合

一方で、懇親会での質問や反応として目立ったのは以下だった。

* そもそもどう動いているのか
* ローカルLLMとは何か
* なぜChatGPT APIではないのか
* どうやってLINEとつないでいるのか
* アバターと音声はどう連動しているのか
* Director / Actorとは何か
* なぜ2つのAIに分けるのか

この差分は非常に重要である。

今後の外向け説明では、自分にとっての当たり前を前提にせず、以下から始める必要がある。

```text
普通のChatGPT利用は、クラウド上のAIにメッセージを送ります。

ジェムちゃんは違います。
LINEは入口だけで、裏側では自宅サーバー上のローカルLLMが動いています。

さらに、1つのAIに全部やらせるのではなく、
文脈を読むDirectorと、キャラクターとして話すActorに分けています。
```

### 解決策

#### 3.8 事前の動画・静止画・台本が発表の保険として機能した

ライブデモには問題が発生したが、49秒のmp4動画をスライドに埋め込んでいたため、完全な失敗にはならなかった。

動画には以下が含まれていた。

* LINE画面
* アバター画面
* ジェムちゃんのダンス
* 音声
* 吹き出し
* Director / Actor構成を示す返答

これにより、少なくとも「動いているジェムちゃん」の姿は観客に伝わった。

また、予備の静止画も用意していたため、動画が動かなかった場合の退避路も存在していた。

印刷台本も、完全には計画通り読めなかったものの、発表骨格を維持する助けになった。

#### 3.9 Director / Actorを「監督と女優」と表現したことで、中核説明は維持できた

埼玉スタジアム2002での地域団体向けイベントにおける事前出展時に、Director / Actor構成を「監督と女優」として50回以上説明していた。

そのため、当日も以下の説明だけは身体に残っていた。

```text
ジェムちゃんは、監督と女優の二つのAIエージェントが協力して動いています。

監督が台本を書いて、女優がそれを元に演じるという役割分担をしているのが特徴です。
```

これは今後も使える説明である。

技術者にも一般参加者にも伝わりやすく、ジェムちゃんのアーキテクチャの中核を一言で表現できる。

#### 3.10 ライブデモ失敗を通じて、次回のデモ運用設計が明確になった

今回の失敗により、次回改善すべき点が明確になった。

次回のLTモードでは、以下を設計する。

```text
- 動画音声はスライド埋め込みで保証する
- ライブアバター音声は無理に出さない
- ライブ時は字幕・吹き出し・表情変化を優先する
- 音声ありデモはローカルmp4で見せる
- Google Meet / 会場音響 / PC出力の音声経路を事前確認する
```

アバターのライブ音声は魅力的だが、画面共有や会場音響が絡む場では事故率が高い。

今後は、音声をライブで必須にせず、確実に再生できる動画側に寄せる方が安定する。

#### 3.11 SSEヘルスチェック機能の必要性が明確になった

今後、アバター画面には以下を追加する必要がある。

```text
- アバター制御SSEの接続状態表示
- ログ表示SSEの接続状態表示
- 最終受信時刻表示
- 手動再接続ボタン
- streamコンテナ再起動手順への導線
- 異常時に画面上で分かる警告表示
```

特に、ログ表示用streamの停止は、画面から気づきにくい。

デモ時には「裏では死んでいるが、画面は一見動いている」という状態が最も怖い。

デモモードでは、内部の動作状態を安全な範囲で可視化する必要がある。

#### 3.12 デモ環境切り替えを設定化する必要がある

今回のようにコメントアウトで切り替える方式は、短期的には対応できるが、継続運用には向かない。

次回以降は以下を行う。

```text
- `.env.demo` と `.env.home` の分離
- API接続先の設定化
- SSE接続先の設定化
- アバター衣装の設定化
- デバッグ表示ON/OFFの設定化
- ローカルIPや内部情報の表示可否を設定化
- デモ用ビルドコマンドの整備
```

これにより、デモ直前の手作業変更をなくし、誤設定リスクを減らす。

#### 3.13 次回は「エンタメ」より「構成解説」寄りにする方針を得た

今回の発表はエンターテイメント方向に振った。

ジェムちゃんのアバター、音声、ダンス、ツンデレボイスは強いインパクトを持っていた。

しかし、自分自身の発表適性としては、以下の方向が向いていると感じた。

```text
- 私はジェムちゃんの解説員である
- 今日はAIエージェントの構成を説明する
- 見た目はキャッチーだが、主題はアーキテクチャである
- 趣味の産物ではなく、ローカルLLMエージェントの設計例として話す
```

このロールを被ることで、緊張を減らせる可能性がある。

「私の趣味の成果を見せる」よりも、「構成を説明する技術者」として立つ方が、自分には合っている。
