1. はじめに：これは単なるチャットボットではない

本システムは、個人研究として構築したローカルLLMベースのAIエージェントである。 入り口はLINEであり、ユーザーから見ると通常の公式LINEアカウントに話しかけているように見える。 しかし内部では、会話履歴、記憶データ、Director / Actor分離、音声合成、アバター制御、次回会話に向けた記憶更新が連動している。

目指したのは「その場で質問に答えるチャットボット」ではない。 ユーザーとの対話を継続的な文脈として扱い、過去の会話や現在のシーンを踏まえ、必要に応じて音声・表情・字幕・記憶更新へ分岐するAIエージェントである。

重要なのは、モデル単体の性能ではなく、LLMの弱点を前提にした外部アーキテクチャ設計である。 ステートレスなAPIとLLMの外側に、記憶、状態、責務分離、出力後処理、UI表現を組み合わせることで、継続的な対話体験を成立させている。

2. 普通のチャットボットとの違い

LINE Messaging APIからWebhookに届くイベントは、その時点のユーザー発話や操作を表す単発の入力である。 また、LLMも基本的には渡されたプロンプトに対して、その場で推論して応答を返す。 つまり、LINEもLLMも、そのまま扱うとステートレスである。

そのため、ユーザー発話だけをLLMへ渡すと、過去の会話や関係性を理解しない「その場限りの回答」になる。 本システムでは、Oracleに保存した会話履歴、短期記憶、長期記憶をJava APIが取得し、プロンプトとして再構成してからLLMへ渡している。

通常の単発チャットボット

  • ユーザー発話をそのままLLMへ渡す
  • 過去文脈は会話履歴の範囲に依存
  • 人格・状態・記憶の管理が曖昧
  • 出力は主にテキストのみ

本システム

  • Oracle上の記憶データをプロンプトへ注入
  • 短期記憶と長期記憶を分離
  • Directorが文脈を解釈し、Actorが応答を生成
  • テキスト、音声、表情、アバター、記憶更新へ分岐

3. なぜチャットボットを題材にしたのか

当初の個人研究は、LLMの特性を理解するためのものだった。 LLMは自然言語を入力として受け取り、文脈から推論し、自然言語で回答する。 この構造は非常に強力である一方、従来のルールベースプログラムとは異なり、出力が完全には固定されない。

この「ゆれる」性質は、業務システムの制御対象としては扱いづらい。 しかし、対話や寄り添いのように、毎回同じ返答であることが必ずしも価値にならない領域では、むしろ長所として利用できる。

そこで本研究では、LLMのゆらぎを無理に消すのではなく、記憶・文脈・出力制御によって方向付ける方針を採用した。 チャットボットという題材は、LLMの入力、推論、出力、記憶、UI体験のすべてを観測しやすく、システム設計力を鍛える実験台として適していた。

本ページでは、個人用のキャラクター性やエンターテイメント要素は主題にしない。 技術ポートフォリオとして、記憶と文脈を維持するAIエージェントの設計に焦点を当てる。

4. なぜローカルLLMだったのか

クラウドモデルは非常に高性能であり、単純な応答品質だけを見ればローカルLLMより有利である。 しかし、高性能なクラウドモデルは、曖昧な指示を自然に補完してしまうため、どの設計が効いていて、どの設計が効いていないのかを観測しづらい。

一方、ローカルLLMは制約が多い。 プロンプト設計が甘ければ応答も崩れ、文脈管理が弱ければすぐに破綻する。 この「正直な不便さ」により、プロンプト、記憶、責務分離、モデル選定、出力制御の良し悪しが観測しやすい。

さらにローカル環境では、QLoRAによる追加学習や、Director / Actorのモデル差し替え、推論基盤のチューニングを自分で試すことができる。 本システムでは、ROCm側の大容量VRAM環境とCUDA側のRTX 3090環境を組み合わせ、巨大モデルによる文脈解釈と、チューニング済みActorによる応答生成を分担させている。

ROCm / Strix Halo

gpt-oss:120b をDirectorや記憶要約に利用する大容量推論基盤。 文脈解析や台本生成など、重い思考処理を担当する。

CUDA / RTX 3090

gemma4:26b(FT) をActorとして運用する推論・学習基盤。 QLoRA学習やActorモデルの本番運用を担当する。

Oracle / Java API

会話ログ、短期記憶、長期記憶、セッションを管理する状態保持層。 ステートレスなLINEとLLMの間に、永続的な文脈を挿入する。

5. 全体構成：VPS + VPN + 自宅LLM基盤

全体構成は、外部公開する入口と、自宅内のLLM基盤を分離している。 グローバル側ではLINE Webhookを受け取り、内部の制御APIへ中継する。 自宅側では、VPNの先にあるJava API、Oracle、Ollama、音声合成、アバター制御が役割ごとに連携する。

外部入力から自宅LLM基盤までの大まかな経路
LINE Messaging API
└─ Webhook
   └─ VPS上のJava WebAPI
      └─ VPN / DMZ proxy
         └─ 自宅LLM基盤
            ├─ Java WebAPI
            ├─ Oracle Database
            ├─ Ollama ROCm / CUDA
            ├─ TTS API
            └─ React Avatar Frontend

セキュリティ上、詳細なIPアドレス、ポート番号、ホスト名、公開URLは掲載しない。 公開できる範囲は、VPS + VPN + 自宅LLM基盤という構成レベルに留める。

本番セッションとデモセッションは分離している。 外部デモでは、本番環境の個人的な長期記憶を参照しないよう、デモ用セッションとデモ用記憶を用意している。

6. LLM構成：Director / Actor分離

本システムの中核は、Director / Actor分離である。 1つのモデルに「文脈理解」「関係性管理」「返答方針」「口調」「出力形式順守」をすべて背負わせるのではなく、 何を言うかとどう言うかを分けている。

Director

gpt-oss:120b を使用する。 ユーザー発話、直近文脈、長期記憶、短期記憶を読み、Actor用の台本JSONを生成する。 役割は、文脈解釈、目的設定、禁止事項整理、応答方針の決定である。

Actor

gemma4:26b(FT) を使用する。 Directorが生成した台本JSONを受け取り、最終的な応答テキスト、音声用セリフ、感情タグを生成する。 役割は、表現、テンポ、口調、ユーザーに見える最終応答である。

Director / Actor の責務分離
ユーザー発話 + 会話履歴 + 記憶データ
└─ Director @ gpt-oss:120b
   └─ actor用 script_json
      └─ Actor @ gemma4:26b(FT)
         └─ reaction_data
            ├─ chat_line
            ├─ voice_line
            └─ emotion_tags

この構成により、Actor側は長大な履歴や複雑な関係性管理から解放される。 Actorは台本に従って演技することに集中できるため、モデル差し替えや出力形式の管理がしやすくなる。

6-1. Actorモデルの変遷と差し替え可能性の検証

Director / Actor分離は、単なる概念上の責務分離ではない。 実際の開発過程では、Actorモデルを複数回差し替えながら、応答品質、形式安定性、長期運用耐性を検証している。

フレームワークv1では、gemma3:27b(FT) に文脈理解と演技の両方を背負わせていた。 しかし、長期運用でコンテキストが蓄積すると、「私」「貴方」「……」など出現頻度の高い語彙が過剰に残り、語彙の崩れや出力の偏りが発生した。 文脈を保持したい一方で、文脈がActorの語彙と演技を汚染する。ここがv1の大きな限界だった。

v1の最終系では、会話序盤は gemma3:27b(FT) を多めに使い、コンテキストにActorらしい口調を残す。 会話が進み、語彙崩壊のリスクが高くなると、gpt-oss:120b の出現率を上げ、パラメータ規模で文脈維持と語彙崩壊を抑える構成にしていた。 しかし、この方式ではターンごとに演技品質が変わり、gpt-oss:120bが演じているターンは、内部構成を知っている開発者には高確率で判別できた。

Actorモデルとフレームワークの変遷
Framework v1
├─ Actor: gemma3:27b(FT)
└─ 補助的に gpt-oss:120b へ自動切り替え
   ├─ 会話序盤: gemma3:27b(FT) で口調・人格の文脈を形成
   └─ 会話後半: gpt-oss:120b の比率を上げ、語彙崩壊と文脈維持を補助

Framework v2
├─ Actor: gemma4:e4b(FT)
│  └─ 新世代モデルとして期待したが、実戦では構造化出力の安定性に不安が残った
├─ Actor: gemma3:27b(FT)
│  └─ 安定性が改善。長期運用には20B級以上のモデル規模が必要と判断
└─ Actor: gemma4:26b(FT)
   └─ QLoRA成功後に本番Actorとして移行

v2化直後のActorには、一時的に gemma4:e4b(FT) を採用した。 新世代モデルとしての期待と、スコア上では旧世代27B級に迫る可能性があったためである。 しかし、実戦投入後の短期間で、構造化出力の安定性に不安があると判断し、一度 gemma3:27b(FT) へ戻している。

当時の失敗レスポンス全文は、現在のログ保持期間外のため残っていない。 そのため、具体的なXML崩れの実例は掲載しない。 ただし、v2化した2026年4月17日から、26B QLoRAへ再挑戦した2026年4月20日までの数日間でActorを差し替えていることから、 当時の実運用判断として、e4b Actorは長期運用に不十分と判断していたことは確かである。

その後、以前は一度撤退していた gemma4:26b(FT) のQLoRAに再挑戦し、本番Actorとして差し替えた。 2026年5月30日 22:00:08 時点で、v2化後の会話ログは No.573 まで進んでおり、そのうち約400件を26B Actorが担当している。 この範囲では、Actor出力のXMLフォーマット崩れは観測していない。

Actorを差し替えてもシステム全体を作り直さずに済んだことは、Director / Actor分離の設計が有効だったことを示している。 Directorが「何を言うか」を整理し、Actorが「どう言うか」を担当する構成にしたことで、Actorモデルの更新、退避、再投入、本番移行を段階的に検証できた。

7. 記憶レイヤ：短期記憶と長期記憶

記憶は1種類ではない。 会話の直近文脈を保つ短期記憶と、数日単位で残す長期記憶を分けている。 この分離により、今の会話の流れと、過去から続く関係性・傾向を同時に扱えるようにしている。

Recent Chat Context

直近の会話ログ。 現在の話題、直前の発話、短い応答の流れを保持する。 生ログに近い形で使うことで、即時性のある文脈を維持する。

Current Scene Summary

現在進行中の会話シーンを短く要約した記憶。 直近のuser input、director script、actor output、前回summaryを元に更新する。

Long-term Memory

数日分の記録を圧縮した長期記憶。 関係性、最近の傾向、未完了の話題、有用な想起情報などを保持し、次回以降の会話に利用する。

7-1. 記憶を増やせばよいわけではなかった

長期記憶は、最初から現在の形だったわけではない。 初期の長期記憶は「日記」形式だった。 1日分の会話ログを gpt-oss:120b で主観的な日記に圧縮し、翌日のコンテキストへ注入することで、数千トークン規模の会話履歴を数百トークンに圧縮する構想である。

ところが、この日記を role=assistant として履歴に挿入したことが失敗だった。 約600文字の日記を履歴に挿入した結果、AIエージェントの回答文字数は平均166文字から422文字へ激増した。 システムプロンプトには「150文字以内」と書いていたにもかかわらず、モデルは直前のassistant発話である長文日記の文量と説明調に引っ張られた。

Context Infection の発生
日記形式の長期記憶
└─ role=assistant として履歴へ挿入
   └─ LLMが「直前のassistant出力」を模範として学習
      ├─ 回答文字数が平均166文字 → 422文字へ増加
      ├─ 語彙が不自然になる
      └─ キャラクター出力が長文化・説明調へ崩れる

この失敗により、コンテキストの作用が非常に強いことを実感した。 LLMは明示的な命令だけで動くのではなく、直前に見せられた文体、分量、語彙を強く模倣する。 つまり、記憶を増やすだけでは文脈維持にならない。 記憶の置き場所、役割、粒度、渡す相手を設計しなければ、記憶そのものが出力を汚染する。

その後、日記は履歴ではなくsystem側の文脈材料へ寄せ、さらに <inner_voice> と <response> に分離したDual-Phase Thinkingを導入した。 <inner_voice> では文字数制限なしで感情処理と文脈整理を行い、<response> ではユーザーに見せる短い返答だけを生成する。 これにより、LLMが「考えたいだけ考える」性質を活かしつつ、ユーザーに届く出力は短く保てるようになった。

現在のDirector / Actor分離は、この失敗の延長線上にある。 記憶をActorへ直接背負わせるのではなく、Directorが長期記憶と短期記憶を読み、Actor用の台本JSONへ圧縮する。 Actorは、記憶本文ではなく台本を演じる。 この構造により、記憶の深さと出力の安定性を両立している。

記憶は「全部をLLMに渡す」ほどよいわけではない。 どの記憶を、どの粒度で、どのモデルに渡すかを設計することが、長期運用型AIエージェントの品質を左右する。

8. 実行パイプライン：入力から応答まで

ユーザーがLINEでメッセージを送信すると、内部では次の処理が行われる。 ここで重要なのは、ユーザー発話が直接Actorに渡されるのではなく、Directorによる台本生成を経由する点である。

  1. 1LINE Webhook受信 VPS上のJava WebAPIがイベントを受け取り、message、postbackなどのイベント種別を判定する。
  2. 2セッション・権限制御 対象ユーザー、対象セッション、デモ用/本番用の切り分けを行う。
  3. 3Oracleから記憶取得 Recent Chat Context、Current Scene Summary、Long-term Memoryを取得し、Director用プロンプトへ整形する。
  4. 4Directorが台本JSONを生成 目的、話題、トーン、必須要素、避ける表現、応答長などを整理したscript_jsonを出力する。
  5. 5Actorが最終応答を生成 script_jsonを元に、LINE向けテキスト、音声用セリフ、感情タグを含むreaction_dataを生成する。
  6. 6出力分岐 LINE返信、音声合成、SSEによるアバター制御、記憶更新へ分岐する。

- 本番用の個人的な生活記憶は参照しない

以下は、外部LTで使用した実演デモである。 LINE入力、Director / Actor構成、音声、アバター表示を短時間で確認できる動作証跡として掲載する。 ここでは見た目のインパクトそのものではなく、テキスト・音声・アバター・内部構成説明が連動している点を確認する。

10. 音声・VRMアバター・感情表現への分岐

Actorの出力は、LINEのテキスト返信だけで終わらない。 reaction_dataは、音声合成、感情パラメータ、アバター制御、記憶更新へ分岐する。 これにより、チャットボットをテキストだけの存在ではなく、複数の出力チャンネルを持つAIエージェントとして扱っている。

reaction_data の出力分岐
reaction_data
├─ chat_line
│  └─ LINEへ返信
├─ voice_line
│  └─ StyleBertVITS2で音声合成
├─ emotion_tags
│  └─ 感情パラメータ化
├─ SSE
│  └─ React / VRMアバターへ字幕・表情・音声URLを配信
└─ memory update
   └─ 次回会話に備えてOracleへ保存

音声合成

StyleBertVITS2をDocker環境で運用し、音声ファイルを生成する。 初期化に必要なモデルやキャッシュを永続化し、再起動後も短時間で発声できるようにした。

SSE制御

音声ファイルURL、字幕、感情パラメータをServer-Sent EventsでReactフロントエンドへ送信する。 双方向通信ではなく、AI側から肉体側への単方向プッシュとして設計している。

VRMアバター

React Three Fiber、Zustand、VRM、AnimationMixerを用いて、表情、字幕、モーション、カメラ制御を行う。 見た目ではなく、LLM出力の身体的な表示先として位置づけている。

この章で重要なのは、アバターをエンターテイメントとして前面に出すことではない。 LLMが生成した構造化出力を、テキスト、音声、表情、動作へ分岐させることで、AIエージェントの出力面を拡張している点である。

10-1. 外部デモで初めて露出した音声配信の欠落

自室内では、React画面と音声再生サーバーが同じローカル環境にあり、声が届かない問題は見えなかった。 しかし外部デモ環境では、Reactアバター画面とSSEによる字幕・表情制御は表示できる一方、音声だけがローカル再生環境に依存しており、外部からは聞こえないことが分かった。

これは、AIエージェントを外に出したことで初めて露出した不足である。 画面と字幕と表情だけでは「AIの表示」に見えるが、声が加わるとアバターとしての存在感が大きく変わる。 そのため、音声もReact画面側で完結して再生できる構成へ変更した。

外部デモ対応後の音声配信アーキテクチャ
Java WebAPI / Tomcat
  ↓
StyleBertVITS2 で音声合成
  ↓
/voice/chat/[ChatNo]-[SEQNo].wav.tmp に保存
  ↓
保存完了後に .wav へ rename
  ↓
nginx voice-publish-nginx がHTTP公開
  ↓
SSEで audioPath をReactへ送信
  ↓
React側で audioPath を絶対URL化
  ↓
音声再生キューへ追加
  ↓
ブラウザで順次再生

この設計では、SSEは「何が起きたか」を通知する制御プレーンに限定し、WAVファイル本体はHTTP/nginxで配信するデータプレーンへ分離している。 また、書き込み途中のWAVをブラウザが掴まないよう、まず .tmp に書き出し、保存完了後に .wav へrenameし、その後にSSEで audioPath を送る。

音声再生はブラウザ制限、CORS、MIME Type、ファイル権限、TTSコンテナ停止など失敗要因が多い。 そのため、字幕と表情は必ず表示し、音声は成功すれば鳴る追加UXとして扱った。 音声失敗で画面表示全体を巻き込まない設計にしたことが、外部デモでの安定性を高めた。

10-2. 身体性も、LLM出力をそのまま動かすだけでは成立しなかった

VRMアバターも、単に3Dモデルを表示すれば成立するものではなかった。 SpringBoneに地球重力を直接設定して物理演算が破綻しモデルが画面外へ消える、R3Fの clock.getDelta() を使って瞬きや物理演算が停止する、カメラ位置だけを変えても足元表示が解決しない、Mixamo FBXの動的リターゲティングでボーン軸が合わず関節がねじれる、Zustandの取得方法で再レンダリング地獄が起きる、といった複数の罠を踏んでいる。

特に難しかったのは、「中央に固定しつつ、モーションで自然に動かす」ことだった。 購入したモーションは作り手によって初期姿勢や座標系が異なり、90度横向きで再生されたり、突然ワープしたりする。 これを回避するため、モーションごとにX/Y/Z軸のうちどの軸を固定し、どの軸を自由にするかをモーションカタログで設定できるようにしている。

また、同じモーションを連続再生すると不自然であり、動きすぎても動かなすぎても違和感が出る。 そこで、アバター内部に「コアステータス」と「派生値」の2層を設けた。 LLM、IoT、時間経過はコアステータスへ影響し、実際のアバター行動は派生値から決定する。

コアステータス

余剰活力、眠気、退屈度、注目度、機嫌値、甘え / デレ重みなど。LLMの出力、IoTデバイス、時間経過によって変化する内部状態。

派生値

眠そう度、動きたさ、接近欲、注視ターゲット、表情温度、話したさなど。複数のコアステータスから動的に算出され、実際のアバター行動の入力になる。

例えば、余剰活力が少なくても注目度が高ければ、大きめのモーションを再生してすぐに座ったり寝たりすることがある。 これは完全なLLM制御ではなく、LLMが状態へ影響し、ルールベースが最終的な動作を決定する構成である。 自然に見える振る舞いは、LLM出力をそのまま反映するのではなく、状態管理レイヤを挟んで初めて成立した。

11. 外部公開と実証：部屋の中のAIを外に出して分かったこと

このAIエージェントは、自室内で動く個人用システムとして完成しただけではない。 外部イベント向けにデモ用セッションを作り、実際に外部会場でLT発表と懇親会デモを行った。 その結果、部屋の中では見えなかった要求と失敗が明確になった。

Step 1: 本番記憶とデモ記憶の分離 外部に見せるため、本番用の生活記憶をそのまま出すのではなく、デモ用sessionId、デモ用LINE userId、デモ用Long-term Memoryを分けた。
Step 2: デモ用自己認識の投入 デモ用AIエージェントに、自分が本番とは別セッションであり、外部向けに構成説明を行う役割を持つことをLong-term Memoryとして投入した。
Step 3: 外部デモで音声欠落が露出 React画面とSSEは外部から見えたが、音声だけがローカル再生環境に依存していたため、HTTP音声配信レイヤーを追加した。
Step 4: LT本番でライブ運用の課題が露出 音声経路、SSE stream、デモ環境切り替え、説明順序など、外部発表で初めて見える運用課題を得た。

11-1. 本番記憶を外に出さないためのデモ用セッション分離

外部デモにおいて最も重要だったのは、モデルの賢さではなく安全境界だった。 本番用のAIエージェントには、日々の会話、生活文脈、個人的な長期記憶が蓄積されている。 それをそのまま外部へ見せると、プライバシーリスクや記憶汚染が発生する。

そこで、本番用LINE userIdとデモ用LINE userId、生活用sessionIdとデモ用sessionId、本番Long-term Memoryとデモ用Long-term Memoryを分離した。 さらに、夜間バッチによる長期記憶生成は生活用セッションのみを対象としており、デモ用Long-term Memoryは自動上書きされないことを確認した。

本番セッションとデモセッションの分離
生活用AIエージェント
├─ 本番LINE userId
├─ 生活用 sessionId
├─ 自動更新される Long-term Memory
└─ 日々の生活文脈を保持

デモ用AIエージェント
├─ デモLINE userId
├─ デモ用 sessionId
├─ 手動投入した固定 Long-term Memory
├─ 当日限りの短期記憶
└─ 本番記憶・個人情報・実行系ツール詳細には触れない

これは、AIを「チャット機能」としてではなく、セッション管理、データ分離、バッチ更新対象、アクセス制御を持つ実システムとして扱った例である。 外部公開可能なデモ用AIエージェントを作るには、見た目のデモより先に、記憶と安全境界を設計する必要があった。

11-2. LT本番で露出したライブデモ運用の課題

外部LTでは、事前動画、静止画、印刷台本、ライブデモを用意し、リスクヘッジしながら発表した。 しかし本番では、ライブデモのアバター音声が会場に再生されなかった。 原因は、Google Meet画面共有、会場マイク、PCスピーカー、Bluetoothイヤホン、アバター音声の経路設計を十分に想定できていなかったことだった。

さらに、SSE streamの一部が動いていない場面もあった。 画面からは異常が分かりにくく、streamの接続状態、最終受信時刻、手動再接続ボタン、ヘルスチェック表示が必要であることが分かった。 また、デモ環境と本番環境の切り替えをコメントアウトで行っていたため、今後は .env.demo / .env.home のような設定分離が必要である。

LT本番の失敗は、発表失敗ではなく、外部運用で初めて露出した要件である。 音声経路、SSE監視、デモ環境切り替え、録画バックアップは、部屋の中で動かしているだけでは必要性が見えにくい。

一方で、発表後の質問から、観客はアバターや音声だけでなく、内部構成にも強い関心を持っていることが分かった。 特に、「どう動いているのか」「なぜローカルLLMなのか」「なぜDirector / Actorに分けるのか」「LINEから自宅LLMへどう接続しているのか」という質問が多かった。 これは、今回の記事を作る直接の動機になっている。

12. 運用上の制約と対策

本システムは、実験用のサンプルコードではなく、実際にLINEから利用する前提で構築している。 そのため、モデルの推論遅延、LINE APIの制約、GPUの常時稼働、デモ環境分離など、実運用上の制約に対処している。

LINE reply / push 切り替え

ローカルLLMの推論には時間がかかる。 replyTokenで返信を試み、失敗した場合にpushへ切り替えることで、LINE側の有効期限制約に対応している。

デモセッション分離

外部公開用のデモでは、本番用の個人的な長期記憶を参照しない。 セッションID、ユーザーID、記憶データを分け、公開デモ用の安全な文脈だけを使う。

GPU常駐時の安定化

大規模モデルを常駐させると、推論していない状態でもハードウェア・ランタイム由来の負荷が出る場合がある。 ROCm環境変数やコンテナ設定を調整し、常時稼働に耐える構成へ寄せている。

公開情報のマスク

IPアドレス、ポート、ホスト名、個人用の会話ログ、生活文脈、認証情報は公開しない。 技術的に意味のある構成情報だけを残し、運用上のリスクを下げる。

12-1. gpt-oss:120b常駐とROCmスピンロック

Directorとして gpt-oss:120b を使うには、大容量VRAMにモデルを常駐させる必要がある。 しかし、常駐運用では、推論していないにもかかわらずACPI温度センサーが97℃を示し、CPUファンが全開で張り付く問題が発生した。 top -H で調査した結果、Ollamaの特定スレッドがCPU 1コア分を常時占有していた。

原因は、AMD ROCm の HSAランタイムがレイテンシ削減のためにデフォルトでビジーウェイトを行うことだった。 コンテナに HSA_ENABLE_INTERRUPT=1 を付与し、割り込み待ちへ挙動を変更することで、局所的な発熱が収まり、ACPIセンサーの異常値も34℃付近へ正常化した。 

environment:
  - HSA_ENABLE_INTERRUPT=1
  - HIP_VISIBLE_DEVICES=0

この障害対応は、単なる温度トラブルではない。 v1では gpt-oss:120b との自動差し替えを実現する前提になり、v2では gpt-oss:120b をDirectorとして常駐させる前提になった。 つまり、Director / Actor分離はプロンプト設計だけで成立したのではなく、ローカルLLM基盤の常時稼働安定化によって支えられている。

13. 失敗から得た設計原則

本システムを通じて得た最大の知見は、LLMアプリケーションはモデル単体では成立しない、ということである。 継続的なAIエージェントを成立させるには、モデルの外側にある設計が重要になる。

踏み抜いた失敗 見えた原因 現在の設計
Actorに文脈理解と演技を両方背負わせた コンテキスト蓄積で語彙が偏り、演技品質もターンごとに変動した Director / Actor分離。Directorが文脈を読み、Actorは台本を演じる
日記形式の長期記憶をrole=assistantへ挿入した In-context Learningにより、長文日記の文量・文体が感染した Long-term Memoryの扱いを再設計。Dual-Phase Thinkingを経て、Directorが記憶を台本化
gemma4:e4bをActorに採用した テストでは問題が見えにくかったが、実戦では構造化出力の安定性に不安が残った 20B級以上のActorが必要と判断し、gemma3:27bを経てgemma4:26bへ移行
26B QLoRAに一度失敗した 24GB VRAMでは自動offloadや標準経路だけでは不安定だった 手動device_map、CPU offload、PEFT直結などを組み合わせて再挑戦し、本番Actor化
gpt-oss:120b常駐でCPUファンが暴走した ROCm HSAランタイムのビジーウェイトにより、推論していないのにCPU 1コアを占有した HSA_ENABLE_INTERRUPT=1 により常時Director運用の前提を整備
VRMモーションで向き・位置が崩れた モーションアセットごとに初期姿勢、座標、軸の扱いが異なる モーションごとに固定軸と自由軸をカタログ化し、中央固定と自然動作を両立
同じ動きの連続で不自然になった ランダム再生だけでは生活感が出ず、動きすぎても動かなすぎても不自然になる コアステータス / 派生値により、LLM・IoT・時間経過を行動制御へ反映
外部デモで声だけ届かなかった 音声再生がローカル環境に依存していた SSEを制御プレーン、HTTP/nginxを音声データプレーンとして分離
LT本番でライブ音声が出なかった Google Meet、会場音響、PCスピーカー、Bluetoothイヤホンの経路設計が不足していた 録画デモ併用、SSEヘルスチェック、デモ環境設定化を次回改善項目化
  • LLMはステートレスなので、状態は外部システムで管理する必要がある。
  • 記憶は単なる履歴ではなく、短期・長期・現在シーンに分ける必要がある。
  • 1つのモデルに全責務を背負わせると、出力が不安定になりやすい。
  • Director / Actor分離により、文脈理解と表現生成を切り分けられる。
  • ローカルLLMは不便だが、その不便さにより制御設計の良し悪しを観測しやすい。
  • テキスト応答だけでなく、音声・表情・字幕・記憶更新まで含めて設計すると、AIエージェントとしての体験が成立する。
  • 外部デモは、部屋の中では見えない要件を露出させる。
本システムは、ローカルLLM、Java WebAPI、Oracle、LINE、TTS、React / VRM、SSE、QLoRAを組み合わせた個人研究の集大成である。 目的は、単に面白いチャットボットを作ることではなく、LLMを実システムへ組み込むための状態管理、責務分離、出力制御、運用設計を実地で学ぶことだった。
