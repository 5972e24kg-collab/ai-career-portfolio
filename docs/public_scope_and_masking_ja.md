# 公開範囲・マスク方針

## 1. この文書の目的

この文書は、本ポートフォリオにおいて公開する情報、公開しない情報、および公開前に行うマスク・抽象化の基準を明確にするための運用方針である。

本ポートフォリオは、個人研究として実施した AI / LLM 技術研究の成果物を、第三者が確認可能な形で提示することを目的とする。

一方で、公開対象にはソースコード、実行ログ、スクリーンショット、動画、設計資料、運用メモなどが含まれるため、業務上の機密情報、個人情報、認証情報、内部システム情報、実運用環境の詳細などを誤って公開しないよう、事前に判断基準を定める。

この文書は、以下の3つの役割を持つ。

1. 自分自身が公開前に確認するためのルール
2. 本ポートフォリオの公開運用ポリシー
3. AIによる公開前レビュー時の基準文書

## 2. 基本方針

本ポートフォリオでは、技術的な取り組み、設計意図、検証過程、失敗と改善、成果物の構造を公開する。

ただし、公開するのは「技術力を説明するために必要な範囲」に限定する。

以下の原則に従う。

- 技術的な説明に必要な情報は公開する
- 機密情報、個人情報、認証情報は公開しない
- 実運用環境を特定できる情報は公開しない
- 業務経験と個人研究を混同させない
- 未確定の実績を、確定済みの成果として表現しない
- 公開対象に迷う情報は、原則として公開しない
- 公開する場合は、マスク・抽象化・サンプル化を行う
- 外部リソース、OSS、AIモデル、データセットのライセンスおよび利用条件を確認する
- AI生成物を公開する場合は、必要に応じてAIによる生成または支援を受けたものであることを明記する
- AIによる査読結果は補助情報とし、最終判断は人間が行う

## 3. 公開する情報

本ポートフォリオでは、以下の情報を公開対象とする。

### 3.1 個人研究として作成した技術資料

- AI / LLM 技術研究の概要
- ローカルLLM基盤構築に関する設計資料
- RAG / ETL パイプラインに関する設計資料
- Director / Actor 型エージェント構成に関する説明
- MCP / Tool Use に関する設計資料
- QLoRA / モデル評価に関する検証記録
- Codex / AI支援開発に関する運用知見
- Work001〜Work076 の索引および要約

### 3.2 公開可能なソースコード

- 個人研究として作成したコード
- 公開用に整理したサンプルコード
- 認証情報や内部接続情報を含まないコード
- 実運用情報を除外したデモ用コード
- README、設計資料、サンプルJSON

### 3.3 マスク済みの証跡

- マスク済みの実行ログ
- マスク済みのスクリーンショット
- マスク済みの動画
- 抽象化した構成図
- サンプルデータを用いた動作例

### 3.4 実利用を想定した成果物の説明

個人開発ではない公的・共同運営イベント、または第三者利用を想定した成果物については、公開可能な範囲に限定して以下を記載する。

- 企画・設計・開発の目的
- 想定利用者
- 要件定義上の制約
- 運用準備の内容
- 実装上の工夫
- UI / ロジック分離などの設計判断
- デモ用に再構成した画面やサンプルデータ

ただし、関係者の個人情報、内部調整内容、未公開の運営情報は公開しない。

### 3.5 AI生成物の扱い

本ポートフォリオでは、AI / LLM を用いた研究過程の一部として、AIが生成した文章、要約、コード、ログ、応答例を掲載する場合がある。

AI生成物を公開する場合は、必要に応じて、AIによる生成または支援を受けたものであることを明記する。

AI生成物には、事実誤認、ハルシネーション、過剰な断定、不適切な推論が含まれる可能性があるため、公開前に人間が内容を確認する。

AI出力をそのまま事実として扱わず、技術的な検証結果、実装内容、実行ログ、人間による判断と区別して記載する。

### 3.6 外部リソース・OSS・AIモデル・データセットに関する情報

本ポートフォリオで外部のOSS、AIモデル、データセット、ライブラリ、ツールを利用している場合は、必要に応じて以下を公開対象に含める。

- 使用したOSS、ライブラリ、ツール名
- 使用したAIモデル名
- 使用したデータセット名
- ライセンス情報
- 公式リポジトリまたは配布元
- 改変の有無
- 再配布可否
- 参考にした資料や実装の出典

ただし、ライセンス上、再配布が許可されていないモデル、重み、データセット、ファイルは、本ポートフォリオに含めない。

## 4. 公開しない情報

以下の情報は、本ポートフォリオでは公開しない。

### 4.1 業務上の機密情報

- 顧客名
- 案件名
- 契約内容
- 業務上知り得た非公開情報
- 社内限定資料
- 内部システムの詳細
- 業務上の障害情報や対応履歴の詳細
- 公開許可を得ていない業務コード

### 4.2 個人情報

- 個人名
- メールアドレス
- 電話番号
- 住所
- 学校関係者、保護者、児童、運営関係者を特定できる情報
- チャットログ内の個人情報
- スクリーンショットに写り込んだ個人情報

### 4.3 認証情報・接続情報

- APIキー
- パスワード
- アクセストークン
- Cookie
- SSH鍵
- `.env` ファイル
- Oracle接続文字列
- JDBC URL
- Portainerの実URL
- Docker / サーバー管理画面の認証情報
- SwitchBot等のIoTトークン
- GitHub token
- その他の秘密情報

### 4.4 実運用環境を特定できる情報

- グローバルIPアドレス
- プライベートIPアドレス
- ホスト名
- サーバー名
- ネットワーク構成の詳細
- ポート開放状況
- 実運用コンテナの完全な一覧
- 実DBスキーマの詳細
- 実データの件数・内容が第三者や業務を特定し得る情報

### 4.5 未確定の実績表現

以下のような、事実確認前または実施前の成果を断定する表現は行わない。

- 成功した
- 多数の利用者に使われた
- 実績を上げた
- 本番運用で高い評価を得た
- イベントで成果を出した

実施前または結果未確定のものは、以下のような表現に留める。

- 出展予定
- 運用準備済み
- 実利用者を想定して企画・設計・開発した
- 公開用デモとして再構成した
- 本番利用前の準備段階である

## 5. マスク・抽象化の基準

公開対象に含まれる情報は、必要に応じて以下の方法でマスクまたは抽象化する。

### 5.1 固有名詞の一般化

固有名詞は、技術説明に必要な場合を除き、一般化する。

例：

```text
特定の会社名 → 現職企業、所属企業、業務システム
特定の学校名 → 主催団体、学校関係組織
個人名 → 運営担当者、関係者、ユーザー
具体的な案件名 → 業務案件、既存システム、連携基盤
```

### 5.2 接続情報の削除

以下の情報は削除または伏字化する。

```text
https://internal.example.local → [internal URL masked]
192.168.xxx.xxx → [IP address masked]
jdbc:oracle:thin:@... → [Oracle connection masked]
Bearer xxxxx → [token masked]
```

### 5.3 ログの抜粋

ログは全文公開しない。

公開する場合は、技術説明に必要な行だけを抜粋し、以下を削除する。

- 実ユーザー情報
- 実データ
- パス
- ホスト名
- IPアドレス
- トークン
- 内部URL
- 業務名
- 個人名

### 5.4 スクリーンショットの加工

スクリーンショットを公開する場合は、以下を確認する。

- ブラウザのURL欄に内部URLが写っていないか
- コンソールにトークンやパスが写っていないか
- 個人名やメールアドレスが写っていないか
- サーバー名やIPアドレスが写っていないか
- チャットログに個人情報が含まれていないか
- デスクトップ上の他ファイル名が写っていないか

必要に応じて、トリミング、ぼかし、塗りつぶし、サンプル画面への差し替えを行う。

### 5.5 動画の確認

動画を公開する場合は、以下を確認する。

- 画面遷移中に内部情報が一瞬でも映らないか
- ブラウザURL、ターミナル、ログ、通知が映らないか
- 音声に個人名や内部情報が含まれていないか
- 字幕に未確認の実績表現が含まれていないか
- ファイルパスに個人情報や内部情報が含まれていないか

必要に応じて、編集、カット、ぼかし、字幕修正を行う。

## 6. 業務経験と個人研究の分離

本ポートフォリオでは、職務経歴書に記載された業務経験と、個人研究として実施した AI / LLM 技術研究を区別して記載する。

業務経験は、職務経歴書に記載済みの範囲で説明する。

個人研究の成果物は、業務実績として誤認されるような表現を避ける。

### 6.1 許容する表現

```text
個人研究として構築した
個人開発として実装した
本業で培ったJava / Oracle / API / 運用保守経験を応用した
業務経験とは別に、AI / LLM技術研究として実施した
```

### 6.2 避ける表現

```text
業務で導入した
会社で運用した
顧客案件として提供した
本番業務で成果を上げた
```

ただし、職務経歴書に記載済みの事実については、その範囲で記載してよい。

## 7. 公的・共同運営イベントに関する表現方針

個人開発ではない公的・共同運営イベント、または第三者利用を想定した成果物については、関係者・運営団体・参加者に関する情報を慎重に扱う。

### 7.1 記載してよい情報

- 公開済みのイベント名称
- 自分が担当した技術的役割
- アプリケーションの設計目的
- 想定利用者
- 技術構成
- UI / UX上の工夫
- 運用準備の内容
- デモ用に再構成した画面

### 7.2 記載しない情報

- 非公開の運営情報
- 関係者の個人名
- 内部連絡内容
- 未公開の会場運用情報
- 参加者の個人情報
- 学校関係者や児童を特定できる情報
- 協賛企業の実名。ただし公開許可がある場合を除く

### 7.3 実施前・結果未確定時の表現

実施前または結果未確定の成果物については、以下の表現に留める。

```text
出展予定
運用準備済み
実利用者を想定して企画・設計・開発した
イベント向けに準備した
本番利用を想定したデモとして公開している
```

以下のような表現は、事実確認後でなければ使用しない。

```text
成功した
多数の参加者に利用された
高評価を得た
イベント運営に大きく貢献した
```

## 8. キャラクター名・研究プロジェクト名の扱い

本ポートフォリオでは、個人研究として構築した対話エージェント、アバター、カスタムモデル、UI、研究プロジェクトに付与した名称については、公開可能な範囲でそのまま使用する。

これらの名称は、個人情報、認証情報、内部URL、実ホスト名、実コンテナ名とは異なり、研究内容を説明するための固有名として扱う。

例：

```text
ジェムちゃん
GemChan
自作AIエージェント名
カスタムモデル名
研究プロジェクト名
UI / ダッシュボード名
```

ただし、以下に該当する場合は、マスクまたは抽象化する。

* 第三者の個人情報と結びつく場合
* 未公開作品、未公開データ、権利上の問題がある素材名に接続する場合
* 家庭内・業務内の実インフラ構成を特定できる場合
* 外部販売物、第三者素材、音声モデル、3Dモデル、モーション素材などの名称を出すことで権利関係が曖昧になる場合
* 公開対象の説明に不要であり、かつ誤解や過度な私的情報の露出につながる場合

技術レポートにおいて、対話エージェントやカスタムモデルの名称が研究対象を説明するうえで必要な場合は、名称を残してよい。

本ポートフォリオでは、名称そのものを秘匿することよりも、認証情報、接続情報、個人情報、実運用環境を特定できる情報を確実に除外することを優先する。

## 9. ソースコード公開時の注意

ソースコードを公開する場合は、以下を確認する。

- 認証情報が含まれていないか
- `.env`、設定ファイル、秘密鍵が含まれていないか
- 実運用URLがハードコードされていないか
- 接続先ホスト名が含まれていないか
- 社内コードや業務コードを流用していないか
- コメント内に内部情報が含まれていないか
- サンプルデータに実データが混入していないか
- READMEに誤解を招く表現がないか
- 使用しているOSS、AIモデル、データセットのライセンスや利用条件に反していないか
- AI生成コードを公開する場合、人間が内容を確認しているか

### 9.1 サンプルデータの扱い

公開用サンプルデータは、原則として新規に作成したダミーデータを使用する。

実データを加工して使用する場合は、個人・組織・案件・環境を特定できない状態にする。

### 9.2 設定ファイルの扱い

設定ファイルを公開する場合は、以下の形式にする。

```text
.env.example
config.sample.json
application.example.properties
```

実際の接続情報を含むファイルは公開しない。

### 9.3 Git履歴の確認

公開用リポジトリに push する前に、現在のファイル内容だけでなく、Git のコミット履歴に機密情報が含まれていないか確認する。

特に、以下の情報が過去のコミットに含まれていないか確認する。

- `.env` ファイル
- APIキー
- パスワード
- アクセストークン
- Cookie
- SSH鍵
- Oracle接続文字列
- JDBC URL
- 内部URL
- IPアドレス
- ホスト名
- 実運用環境の設定ファイル
- 実データを含むログやサンプルファイル

公開前に機密情報の混入が見つかった場合は、その履歴を含むブランチを公開しない。

必要に応じて、公開用に新しいリポジトリまたは新しい履歴を作成し、機密情報を含まない状態で初回コミットを作成する。

Public リポジトリへ push した後の履歴修正は、完全な削除を保証できないため、原則として「push 前に検出する」ことを重視する。

### 9.4 外部リソース・OSS・AIモデルの扱い

本ポートフォリオで外部の OSS、AIモデル、データセット、ライブラリ、ツールを利用している場合は、それぞれのライセンス、利用規約、配布条件を確認する。

公開時には、必要に応じて以下を明記する。

- 使用したOSS、ライブラリ、ツール名
- 使用したAIモデル名
- 使用したデータセット名
- ライセンス情報
- 公式リポジトリまたは配布元
- 改変の有無
- 再配布可否
- 参考にした資料や実装の出典

ライセンス上、再配布が許可されていないモデル、重み、データセット、ファイルは、本ポートフォリオに含めない。

第三者のコードを参考にした場合は、必要に応じて出典、ライセンス、参考範囲を明示する。

## 10. AI査読時の利用方法

本方針は、人間による公開判断だけでなく、AIによる公開前レビューの基準文書としても利用する。

AIに査読させる場合は、以下のような指示を与える。

```text
あなたは、セキュリティ、コンプライアンス、技術文書公開の観点から公開前レビューを行う専門家です。

提供された「公開範囲・マスク方針」の全ルールに厳密に従い、次の文書、コード、ログ、README、スクリーンショット説明文、動画説明文に違反がないか確認してください。

特に以下を重点的に確認してください。

- 個人情報
- 業務上の機密情報
- 認証情報
- 内部URL
- IPアドレス
- ホスト名
- 実運用環境の詳細
- Git履歴に残る可能性のある機密情報
- OSS、AIモデル、データセットのライセンス表記不足
- AI生成物であることの明記不足
- AI出力を事実として断定している表現
- 未確定の実績表現
- 個人研究を業務実績と誤認させる表現
- 公的・共同運営イベントに関する過剰な具体情報
- スクリーンショットやログに含まれる公開不適切情報

違反の可能性がある箇所を、理由と修正案付きで列挙してください。
判断が難しい箇所は「要人間確認」として分類してください。
```

AI査読は補助的な確認手段であり、最終的な公開判断は人間が行う。

## 11. 公開前チェックリスト

公開前に、以下を確認する。

- [ ] 業務上の機密情報が含まれていない
- [ ] 個人情報が含まれていない
- [ ] APIキー、トークン、パスワードが含まれていない
- [ ] 内部URL、IPアドレス、ホスト名が含まれていない
- [ ] 実運用環境を特定できる情報が含まれていない
- [ ] 公的・共同運営イベントの関係者情報が含まれていない
- [ ] 未確定の成果を断定していない
- [ ] 個人研究を業務実績と誤認させていない
- [ ] ログは必要範囲のみ抜粋している
- [ ] スクリーンショットは必要に応じてマスクしている
- [ ] 動画に内部情報が映り込んでいない
- [ ] READMEや説明文が誇張表現になっていない
- [ ] 公開する意味がある情報に限定されている
- [ ] Gitの過去コミット履歴に機密情報が含まれていない
- [ ] OSS、AIモデル、データセットのライセンスや利用条件を確認している
- [ ] 必要なライセンス表記・出典表記を行っている
- [ ] AI生成物であることを必要に応じて明記している
- [ ] AI出力を未検証の事実として扱っていない

## 12. 更新方針

この文書は、本ポートフォリオの公開運用における基本方針である。

頻繁に変更するものではないが、公開対象の種類が増えた場合、または新しいリスクが見つかった場合には更新する。

更新する場合は、変更理由を明確にし、可能な限り既存の公開判断に影響が出ないようにする。

## 13. 最終判断の原則

公開可否に迷う情報は、原則として公開しない。

公開する場合は、以下のいずれかを満たすことを確認する。

- 技術説明に必要である
- 第三者が成果物を理解するために必要である
- 既に公開されている情報である
- 十分にマスク・抽象化されている
- 公開しても個人、組織、業務、運用環境を特定できない
- ライセンス、利用規約、再配布条件に反していない
- AI生成物である場合、人間による確認を経ている

本ポートフォリオでは、技術力の証明と同時に、安全な公開運用を重視する。
