1. はじめに：これは「AIにコードを書かせた話」ではない

このページは、SwitchBot制御用MCPサーバー2号機を、ChatGPT、Claude、CodexなどのクラウドAIを併用して構築した記録である。

ただし、本ページの主題は「AIにコードを書かせたらMCPサーバーが完成した」という単純な成功談ではない。

今回の開発では、私はあえて自分でコードを書かなかった。

MCPサーバーの基盤も、SwitchBot API呼び出しクラスも、既に自作済みだった。 単に完成させるだけであれば、自力で実装した方が短時間で済んだ可能性が高い。

それでもCodexを使った理由は、開発速度を上げるためではない。

目的は、AIを使って低コストにコードを量産することではなく、AIを設計・分析・実装・レビューのプロセスへ組み込み、 人間がGo / NoGo判断と最終責任を持つ開発スタイルを検証することだった。

今回得られた最大の成果は、ソースコードそのものではなく、 AIを使って初期投資コストを上げ、中長期的な保守性・安全性・判断可能性を得るという開発プロセスの実体験である。

2. 今回作ったもの

今回作成したのは、SwitchBotのシーン操作をMCP Toolとして公開する小型MCPサーバーである。

scenes

MCPサーバーから利用可能なSwitchBotシーン一覧を返すTool。 SwitchBotアプリに存在する全シーンではなく、MCP公開許可済みのシーンだけを返す。

executeScene

scenesで返されたsceneKeyを指定して、対応するSwitchBotシーンを実行するTool。 sceneIdはLLMに直接扱わせない。

重要なのは、LLMにSwitchBotのデバイスを直接操作させない点である。

家電や照明といった物理世界の操作をLLMに任せる場合、操作対象や責任境界を曖昧にしてはいけない。 そのため、今回のMCPサーバーでは、SwitchBotアプリ側で人間が定義した「シーン」のみを実行対象とした。

さらに、SwitchBot APIが返すsceneIdをLLMへ直接渡すのではなく、 人間が管理するsceneCatalog.jsonに定義されたsceneKeyを通して操作する設計にした。

3. なぜ、自分で書かずにCodexを使ったのか

今回の開発は、自分で実装できないからAIに頼ったものではない。むしろ逆である。

MCPサーバーの共通基盤は、既にPortainer向けMCPサーバー1号機で構築済みだった。 SwitchBot APIを呼び出すJavaクラスも、日常的にStream Deckから家電操作を行うために実運用していた。

つまり、今回のSwitchBot MCPサーバーは、既存資産を接続すれば完成する見込みがあった。 自力で作るだけなら、おそらく短時間で終わった。

しかし、それでは今回の目的にはならない。

今回検証したかったのは、AIがコードを書けるかどうかではない。 AIを実務的な開発プロセスに組み込んだとき、人間はどこで判断し、どこで止め、どこで採用し、どこで責任を持つべきかである。

4. AIコーディングには2種類ある

今回の作業を通して、私はAIを使ったコーディングには少なくとも2種類あると感じた。

バイブコーディング

AIを使って低コストでソースコードを出力する開発。 UIの試作、デザイン案の量産、短期間で複数パターンを比較したい場面では非常に強い。

監督責任型AI開発

AIを使って設計・分析・実装・レビューの密度を上げる開発。 人間がGo / NoGo判断と最終責任を持ち、中長期的な保守性・安全性・判断可能性を得る。

Work079は後者の実践だった。

5. 今回の開発体制：複数AIを役割分担させる

今回の開発では、複数のAIを同時に使った。

ChatGPT 5.5

仕様相談、方針策定、分析プロンプトの作成、Codexの回答評価、Go / NoGo判断の整理を担当。 開発総合本部として扱った。

Claude Code Opus 4.7

細かい用語の確認、考えの壁打ち、仮作成したプロンプトの評価、別視点でのレビューを担当。

Codex / ChatGPT 5.5

IntelliJ IDEA上で実際にコードを生成する役割。 分析フェーズと製造フェーズを分けて依頼した。

Gemini 3.1 Pro

必要に応じて別系統の確認や補助的な調査に使用した。

ここで重要なのは、単に複数のAIを並列に使ったことではない。 それぞれのAIに役割を持たせ、文脈の汚染を防ぎながら、必要な情報だけを主軸の開発チャットに戻したことである。

6. 実際の開発フロー

今回の開発フローは、おおむね次のような流れだった。 

1. ChatGPTと仕様・方針を相談する
2. Claudeで用語確認や考えの壁打ちを行う
3. ChatGPTで分析用プロンプトを作成する
4. Codexで分析タスクを実行し、分析レポートを取得する
5. 私が分析レポートを読み、判断すべき点にコメントする
6. 必要に応じてClaudeへ相談する
7. ChatGPTへ分析結果と私の判断を戻し、Go / NoGoを決める
8. Goの場合、ChatGPTで製造用プロンプトを作成する
9. Codexで製造を実行する
10. Claudeにソースコードをレビューさせる
11. 私自身もソースコードを全文読む
12. 自分のPCで動作確認する
13. 結果をChatGPTへ戻し、次の方針を立てる

特に意識していたのは、ChatGPTとClaudeの使い分けである。

私にとって、技術開発を前に進める主軸はChatGPTである。 そのため、ChatGPTのチャットには、後から開発経緯を振り返るために必要な情報だけを残すようにしている。

このチャットは長くなることが最初から分かっていた。 さらに、プロジェクト間でメモリや文脈が共有される場合もある。 だからこそ、主軸のチャットには、できるだけ純度の高い情報だけを残したかった。

開発中には、思わずガッツポーズしたくなる場面もある。 しかし、私はChatGPTに「やったね！」のような感情的な反応をあまり書かない。 それは、技術開発の文脈上、後から必要になる情報ではないからである。

同じ理由で、「gitのコマンドを忘れた」「dry-runとは何か」といった、その場限りの疑問も、基本的にはChatGPTに投げないようにした。 それらは開発中の私にとっては必要な疑問である。 しかし、Work079という技術開発の主軸から見ると、必ずしも中心的な情報ではない。

今回、dry-runという言葉が出てきたとき、私は実は初めてその用語を意識した。 その場で「ChatGPTがdry-runと言っているけど、これは何か」とClaudeに相談した。 これはChatGPTを信頼していないからではなく、ChatGPTを主軸として大切に扱っているからである。

ChatGPTのチャットは、開発の本流を記録する場所である。 そこに、すべての小さな疑問や感情や寄り道を流し込むと、文脈が濁る。 長い開発では、この濁りが後半の判断精度に影響する可能性がある。

一方で、Claudeで確認した内容であっても、後の技術レポートや設計判断に必要だと判断したものは、あえてChatGPTへ戻す。 なぜなら、主軸であるChatGPTのチャットに残っていなければ、最終的な技術レポートの材料として扱いにくくなるからである。 

ChatGPT:
  開発の主軸
  仕様・判断・経緯・成果を残す場所

Claude:
  横道の確認
  用語調査
  思考整理
  別視点レビュー

Codex:
  実装担当

Gemini:
  補助的な確認・別視点

複数AIの使い分けは、単なる便利技ではない。 それは、文脈の純度を管理し、判断ログを整理し、AIを開発プロセスの中で安全に使うための運用設計である。

7. Go / NoGo判断が品質を作る

今回の開発で最も重要だったのは、Codexに製造させることではなく、製造に進む前に止める判断だった。

NoGo SwitchBotApisV2 初回分析

Codexの分析自体は優秀だったが、背景情報が広すぎた。 低レイヤーAPIクライアントにMCP固有の責務が混入する危険があったため、スコープを切り直した。

NoGo SceneService 初回分析

cooldownとdry-runが抜けていた。 McpServletではなくSceneServiceへ実行制御を寄せる判断により、責務分離を維持した。

AIの分析結果は優秀である。 しかし、優秀な分析結果をそのまま採用することと、良い開発をすることは同じではない。

AIが出したものを読み、人間が止める。 スコープを切り直す。 必要なら前工程に戻す。

このGo / NoGo判断こそが、今回の品質を作った。

8. 安全設計：LLMに物理世界を触らせるための境界線

SwitchBot MCPサーバーは、LLMから物理世界へ操作を届ける仕組みである。 そのため、安全設計は最初から重要だった。

今回の基本方針は、デバイス直接操作を禁止し、SwitchBotのシーン実行に限定することだった。 

SwitchBot APIのgetScenesに存在する
かつ
sceneCatalog.jsonにも存在する
かつ
sceneIdが一致する
かつ
sceneCatalog.json側のexecutableがtrue

このinner join方式により、SwitchBot側に存在するだけのシーンはLLMへ公開されない。 人間がsceneCatalog.jsonに登録し、公開してよいと判断したものだけがSceneSummaryとしてMCPへ出る。

また、管理用のnoteはLLMに渡さない。 実行時も、LLMはsceneIdではなくsceneKeyを指定する。

これに加えて、同じsceneKeyの連続実行を抑止するcooldownを入れた。 さらに、実際にはSwitchBot APIを呼ばずに解決処理だけを確認できるdry-runモードも用意した。

9. プロンプトは設計書であり、判断ログである

今回公開するプロンプト群は、単なるAIへの命令文ではない。

それらは、分析対象、スコープ、禁止事項、責務境界、Go / NoGo条件を明文化した設計書である。

分析フェーズのプロンプトでは、必ず「コード変更・ファイル作成・実装は行わない」と明記した。 これは、いきなりAIに実装させないためである。

製造フェーズでは、前工程で合意した仕様、作成してよいファイル、変更してよい範囲、変更してはいけない範囲を明示した。 

分析する
判断する
製造する
確認する
次へ進む

完成したソースコードだけを見ても、なぜその設計になったのかは分からない。 しかし、プロンプトと分析レポートとGo / NoGo判断を合わせて見ると、開発の判断過程が見える。

10. この開発方法は楽ではない

今回の方法は、楽ではない。 AIを使えば開発が楽になる、という単純な話ではなかった。

私はChatGPT、Claude、Codexの返答をほぼすべて読んだ。 しかも、多くの場合は2回読んだ。

1回目は全体を把握するため。 2回目は、自分が判断すべき点を拾い、回答するためである。

AIが出した分析やコードを読まずに採用するなら、それは監督責任型の開発ではない。 人間が判断責任を持つなら、読むしかない。理解するしかない。疑うしかない。

この方法は、実装時間を減らすかわりに、認知負荷を増やす。 ただし、その認知負荷は、設計品質、判断可能性、保守性、そして後から説明できる開発ログへ変換される。

11. AIのコードは美しい。しかし、人間には高カロリーである

Codexが出力するソースコードは美しい。 構造は整理されており、責務も分かれており、Javaとしても洗練されている。

しかし同時に、人間が保守するには高カロリーだとも感じた。

AIが書くコードは、密度が高い。無駄が少ない。余白が少ない。 それは品質としては良いことでもある。 しかし、人間が読むときには、少し息継ぎが欲しくなる。

私のコードは、もっと冗長である。 同じことを少し丁寧に書く。 処理のまとまりごとに、理解の区切りを置く。

それは、機械的な美しさでは劣るかもしれない。 しかし、人間にとっては「ここまでは理解できた。次へ進もう」というしおりになる。

AIが書いたコードを採用するということは、そのコードを将来の自分が保守するということである。 この事実を忘れてはいけない。

12. Work079で得た結論

Work079で得た結論は、AIコーディングには複数の形があるということだ。

AIを使って低コストでコードを量産する開発は、確かに有効である。 UI試作やデザイン案の比較、短期間で複数パターンを作る場面では、大きな力を発揮する。

しかし、実務的な品質、安全性、保守性、責任境界が必要な開発では、それだけでは足りない。

今回行ったのは、AIを使って初期投資コストを上げる開発だった。

私は今回、AIを使って初期投資コストを上げた。 そのかわりに、中長期的な保守性、安全性、判断可能性、そして開発プロセスそのものの再現性を得た。

これは、バイブコーディングではない。 AIを開発プロセスに組み込み、人間が監督責任を持つ開発である。
