## 背景

このクラスは、SwitchBot API を呼び出すための既存クラスです。

現在は個人利用の範囲で継続利用している最小構成のAPIクライアントであり、主に以下の機能を持っています。

- SwitchBotのデバイス一覧取得
- SwitchBotのシーン一覧取得
- SwitchBotのシーン実行

今後、このクラスを自作MCPサーバーに組み込みます。

MCPサーバーでは、LLMから以下のような操作を行う想定です。

- 「照明系シーンを実行して」→ SwitchBotの照明系シーンを実行
- 「照明系シーンを停止して」→ SwitchBotの消灯系シーンを実行
- 「照明系シーンをくつろぎモードにして」→ くつろぎモードの照明系シーンを実行
- 「空調系シーンを停止して」→ 空調停止シーンを実行

ただし、MCPサーバーから直接デバイス操作は行いません。  
操作対象は **SwitchBotのシーンに限定**します。

理由は以下です。

- デバイス直接操作は、LLMが触るべきでない機器を操作する危険がある
- 「照明系」などの曖昧語が、部屋や用途によって分岐する
- 実際にどの機器をどう動かすかは、SwitchBotアプリ側のシーン定義に閉じ込めたい
- MCPサーバーの責任は「ユーザー意図に合うシーンを実行する」までに限定したい

## 現状コードで把握してほしい観点

以下の観点で分析してください。

### 1. 現在の機能

現在の `SwitchBotApis.java` が持っている機能を整理してください。

特に以下を確認してください。

- `getDevices()`
- `getScenes()`
- `execScene(String sceneId)`
- `doGet(String uri)`
- `doPost(String uri)`
- `Apis` 内部クラス

### 2. 認証処理

SwitchBot API v1.1 の認証処理として、以下がどのように実装されているか確認してください。

- token
- secret
- nonce
- timestamp
- HmacSHA256
- Base64署名
- HTTPヘッダー
  - `Authorization`
  - `sign`
  - `nonce`
  - `t`

### 3. 問題点

リファクタリング前に、現状コードの問題点を整理してください。

想定している問題点は以下ですが、他にもあれば追加してください。

- token / secret がソースコードにべた書きされている
- `doGet` と `doPost` に重複が多い
- `HttpClient` を毎回生成している
- 例外時に空文字を返しており、呼び出し側が失敗理由を判断できない
- `System.out.println` によるログ出力が残っている
- レスポンスの扱いが raw string 中心で、後続のMCP DTO化に向けた整理が不足している
- URI組み立て、APIパス管理、HTTPメソッド指定の責任がやや混在している

### 4. リファクタリング方針

後続工程で `SwitchBotApisV2.java` を作成するとした場合の方針を提案してください。

ただし、この時点では実装しないでください。

方針では以下を含めてください。

- 認証情報の外出し方法
  - 環境変数
  - コンストラクタ注入
  - propertiesファイル
  - どれが適切か
- GET / POST 共通化の設計案
- `HttpClient` の扱い
- 例外設計
- 戻り値設計
- raw response を残すべきか、DTO化するべきか
- 後続のMCPサーバー連携を見据えたクラス責任の分離案

### 5. 互換性維持

現在の利用箇所を壊さないために、どの外部仕様を維持すべきか整理してください。

特に以下を検討してください。

- `getScenes()` は当面 `String` を返すままにするべきか
- `execScene(String sceneId)` は当面 `String` を返すままにするべきか
- いきなりDTO化せず、V2ではまずAPIクライアントとしての整理に留めるべきか
- MCP用DTO変換は別クラスに分けるべきか

### 6. 安全性

このクラスは、将来的にLLMから間接的に呼び出され、SwitchBotアプリに登録済みのシーンを実行する想定です。

その前提で、最低限考慮すべき安全策を整理してください。

観点例：

- sceneId のバリデーション
- 許可済みシーンだけを実行する仕組み
- dry-run モード
- cooldown
- ログ出力時のマスク
- 誤実行時に追跡できる監査ログ
- MCPサーバー側で持つべき責任と、SwitchBot APIクライアント側で持つべき責任の切り分け
