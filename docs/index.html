3分で分かる私

LLMの非決定性を単に抑え込むのではなく、「制御された自由」として設計し、自律的に振る舞うAIエージェントを、アプリケーションからインフラまで一人で構築・運用してきたエンジニアです。

私の全体像

私は、LLMを利用したプロダクトの発案、企画、要件定義から、アプリケーション設計、実装、インフラ、ネットワーク、セキュリティ、障害対応、継続運用までを一貫して行ってきました。

既存の技術を組み合わせるだけではなく、実現したいユーザー体験から必要な機能を逆算し、不足する技術や知識を自ら獲得しながら、実際に動くシステムへ落とし込むことを得意としています。

個人研究では、記憶、文脈、人格、自律的な発話、音声、3Dアバターを統合したAIエージェントを構築しました。日常的に利用する実運用環境として継続稼働させ、モデルの出力品質だけでなく、可用性、障害時の縮退、運用負荷、ユーザー体験まで含めて改善を続けています。

1. LLMアプリケーションの信頼性・可用性設計

LLMの出力は非決定的であり、同じ指示でも常に同じ形式や品質が得られるとは限りません。

私は、その不確実性をモデルだけに解決させるのではなく、出力を受け取るアプリケーション層に、構造検証、修復、再生成、フォールバック、縮退運転を実装しました。

実地運用では、観測された構造化出力の崩れ約14％をアプリケーション側で補正し、ユーザーから見えるエラーを0％に抑えました。

また、方針決定を担うDirectorと、表現生成を担うActorを分離するPlanner–Executor型の構成を採用しています。モデルごとの能力を見極め、適切な責務だけを持たせることで、長期運用における人格や応答品質の崩れを抑えています。

2. 記憶・文脈・自律性を持つAIエージェント設計

ステートレスなLLMに対して、長期的な関係性と直近の会話の流れを維持するため、異なる役割を持つ二つの記憶系統を設計しました。

長期記憶（Long-term Memory）では、会話ログを長期視点で監査し、未完了の話題、ユーザーが覚えておくよう求めた内容、今後の会話に影響する情報などを抽出し、重要度を付けて永続化します。

短期記憶（Scene Summary）では、直近の会話に焦点を当て、会話の進行に合わせて要約を更新します。新しい話題を強く残し、時間が経過した話題を徐々に薄めることで、自然な忘却を含む会話の流れを作っています。

さらに、要約や長期記憶の誤りだけで文脈が決まらないよう、直近の会話原文もプロンプトへ含める安全措置を設けています。

この記憶設計と、Director、Actor、Analyzerなどの役割分担を組み合わせることで、一貫性を維持しながらも、AI自身が話しかけるか、沈黙するか、どのような表現を選ぶかという部分には確率的な自由を残しています。

3. ローカル／オンプレLLM基盤の構築・運用

CUDAとROCmという異なるGPU基盤上で、複数のLLMを役割分担させて運用してきました。

情報や事例が比較的少ないROCm環境でも、現象の観測、原因の仮説化、切り分け、検証を行い、自ら障害を解決しています。

例えば、ROCm上でEmbedding処理がCPUへフォールバックし、実用的な時間で処理できなかった際には、ベクトル計算だけをCUDAノードへ分離する構成へ変更しました。単一のマシンや技術ですべてを解決しようとせず、処理特性に応じて計算資源を配置しています。

また、モデルをVRAMへ常駐させるとアイドル時にもCPU負荷とファン回転が継続する問題について、ドライバの問題と決めつけずに調査しました。その結果、HSAランタイムのビジーウェイトを原因として特定し、割り込み待ちへ変更することで解消しました。

構築だけでなく、監視、バックアップ、ノード分散、障害時の切り替えまで含めて、継続運用できる基盤として設計しています。

4. QLoRAによる用途特化モデルの開発・運用

自作した学習データを用いて、Llama系、Gemma 3系、Gemma 4系モデルのQLoRAを行ってきました。

学習は一度成功すれば終わりではありません。生成結果を観察し、学習の効き方、口調の変化、指示追従性、過学習、表現の偏りを確認しながら、データの内容や配分を調整しています。

限られたVRAM環境でモデル全体を通常の構成では展開できなかった際には、推論時のメモリ使用量と学習時のメモリ使用量が異なることを実体験として理解しました。その上で、今回の演出面への適応という要件に合わせ、学習対象をモデル後半の一部へ限定する構成を選択し、学習を完走させました。

学習後は、LoRAアダプターの作成だけで終わらせず、ベースモデルとのmerge、GGUF変換、量子化、Ollamaへの登録、実際のAIエージェントへの組み込みまで一連の工程を行っています。

現在も、自分の要件に合わせて学習した独自Actorモデルを、実運用するAIエージェントの一部として使用し続けています。

本ポートフォリオにおけるAI開発スタイル

私のAI開発スタイルは、いわゆるバイブコーディングではなく、 Human-in-the-loop と AI Governance を組み込んだ AI-Augmented Software Engineering です。

この3語は、本サイト内で扱う「バイブコーディングではないAI開発」を理解するための概念アンカーです。 AIを単なるコード生成補助としてではなく、設計・実装・検証・文書化・公開判断までを含む ソフトウェア開発プロセスの一部として扱う姿勢を表しています。

AI-Augmented Software Engineering （AIASE / AI拡張型ソフトウェアエンジニアリング）
単にAIでコードを書かせるのではなく、要件定義、設計、実装、レビュー、テスト、 リファクタリング、ドキュメント化、運用改善まで、ソフトウェア開発ライフサイクル全体を AIで拡張する考え方です。
Human-in-the-loop （HITL / 人間介入型）
AIや自動化システムの処理の中に、人間による確認、判断、修正、承認のポイントを 意図的に組み込む設計です。本ポートフォリオでは、AI生成物をそのまま採用せず、 Git branch、レビュー、テスト、公開判断を通じて人間が責任境界を保持しています。
AI Governance / Secure Coding （AIガバナンス / AI時代のセキュアコーディング）
AIを使う上でのルール、責任、リスク管理、監査、説明可能性、セキュリティを含む統制です。 本ポートフォリオでは、MCP Toolの権限制御、JSON Schema検証、公開範囲・マスク方針、 最小権限設計を通じて、AI利用を安全かつ検証可能に扱うことを重視しています。
